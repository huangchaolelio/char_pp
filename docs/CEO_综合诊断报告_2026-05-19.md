## 三、技术深度诊断

### 3.1 架构评分：8/10 ✅ 亮点突出，但需防范分布式单体

**亮点：**
- 五队列系统直接映射业务阶段（扫描→预处理→分类→KB提取→诊断），不是盲目拆分
- FastAPI async + Celery async 双层异步，IO 和 CPU 任务分离
- 业务阶段双列（`business_phase` + `business_step`）通过 ORM `before_insert` 钩子自动派生，单一事实来源
- Feature-016 预处理前置设计解决了大视频 OOM 和重复下载问题

**风险：**
- ⚠️ 分布式单体风险：Worker 直接读写同一 PostgreSQL，缺少防腐层
- ⚠️ API 层是否用了 `AsyncResult.get()` 同步阻塞需审计

### 3.2 代码质量评分：7/10 ✅ 类型注解好，但函数过长

**亮点：**
- 类型注解覆盖率 95.6%（函数级），远超行业平均
- 模块组织清晰：api/routers → services → models → workers 分层合理
- CI Linter（API 命名 + 错误码）作为硬性前置

**问题：**
- 🔴 `api/routers/tasks.py` 1300 行——严重违反单一职责
- 🟡 `services/curation/curation_service.py` 706 行
- 🟡 `services/diagnosis_service.py` 558 行
- 33 处魔法数字散落在 21 个文件中（timeout/retry/max_*）
- 日志使用不一致：部分模块用 `print()` 而非结构化 logging

### 3.3 数据模型评分：8/10 ✅ 设计规范，迁移有序

**亮点：**
- 29 张表，20 次 Alembic 迁移，与 Feature 一一对应
- 索引策略合理：`coach_video_classifications` 6 个索引、`teaching_tips` 5 个索引
- UniqueConstraint 使用恰当（15 个分布在关键表）
- Feature-019 的 per-category KB 复合主键 + partial unique index 设计精巧
- 时区治理彻底：`TIMESTAMP(timezone=False)` + `timezone('Asia/Shanghai', now())`

### 3.4 异步任务系统评分：8/10 ✅ 设计成熟

**亮点：**
- 五队列物理隔离 + 独立并发控制（classification=1, kb_extraction=2, diagnosis=2, preprocessing=3）
- DB 级幂等提交：partial unique index + pg_advisory_xact_lock
- 孤儿任务自动恢复：Worker 启动时 sweep 840s 超时任务
- 批量提交部分成功语义（前 K 条 ACCEPTED，后 M-K 条 QUEUE_FULL）
- KB 提取门槛：ClassificationGateService 校验 `tech_category != 'unclassified'`

### 3.5 API 设计评分：9/10 ✅ 企业级标准

**亮点：**
- 统一 SuccessEnvelope / ErrorEnvelope 响应
- 38 个 ErrorCode 枚举 + ERROR_STATUS_MAP + ERROR_DEFAULT_MESSAGE 三元同步
- 分页参数硬约束（`page_size` 上限 100，越界返回 422 而非静默截断）
- 枚举归一化三件套：normalize → parse → validate
- 直接物理删除策略（不下线哨兵路由，干净）
- OpenAPI/Swagger 自动生成

### 3.6 技术分类规则评分：4/10 🔴 核心瓶颈

- 21 类规则全是定性描述（"手腕快速内收"），缺乏 CV 可消费的量化特征
- 缺少关键帧序列定义（引拍结束→击球瞬间→随挥结束）
- 边界混淆（正手拉弧圈 vs 正手攻球仅靠"摩擦/撞击"区分）
- **这是系统准确率的天花板——规则不量化，模型永远不可靠**

## 五、CEO 决策建议

### 5.1 优先行动路线图

```
第 1–2 周：消化技术债务（止血）
  ├─ 修复 Feature-006 大文件内存瓶颈（2 人周）
  ├─ 修复 KB 流水线 P0 数据可靠性（3 人日）
  └─ 补齐报告模块测试 + UI 组件清理（3 人日）

第 3–6 周：交付高价值 Feature
  ├─ Feature-017 批量处理（依赖 006 修复）
  └─ Feature-018 导出增强（低复杂度，快速交付）

第 3–8 周（后台并行）：技术分类规则量化专项
  └─ 3 人周，将 21 条定性规则翻译为骨骼点时间序列约束
```

### 5.2 战略建议

1. **暂不拆分微服务**：当前五队列架构对业务规模足够。优先巩固模块边界（防腐层 + 数据所有权），而非引入分布式复杂度。

2. **立即启动规则量化项目**：这是系统准确率的根基。没有量化特征，再多 Feature 都是空中楼阁。

3. **建立监控面板**：Celery 任务执行时间、成功率、通道饱和度需要可视化。当前缺少可观测性基础设施。

4. **安全审计**：检查 COS 密钥和 LLM API key 是否有硬编码泄露风险，确认 CORS 配置。

5. **Feature-021 多语言暂缓**：当前用户 94% 中文环境，ROI 低。
