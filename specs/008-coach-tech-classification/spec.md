# 功能规范: 教练视频技术分类数据库

**功能分支**: `008-coach-tech-classification`
**创建时间**: 2026-04-21
**状态**: 草稿
**输入**: 用户描述: "对cos的COS_VIDEO_ALL_COCAH路径下的内容,按教练生成所教学的技术,生成技术分类数据库.技术分类要精细,比如正手劈长和正手拉球要区分.保存相关路径等信息到数据库,方便后续批量提取知识库.本次不涉及知识库提取"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 扫描 COS 目录并生成教练技术分类 (优先级: P1)

系统管理员触发一次扫描任务，系统自动遍历 COS 中 `COS_VIDEO_ALL_COCAH` 路径下所有教练课程目录，识别每个视频所教学的乒乓球技术类别（精细到"正手劈长"、"正手拉下旋"等），并将结果保存到技术分类数据库。

**优先级原因**: 这是整个功能的核心，没有分类数据就无法支撑后续批量知识库提取。

**独立测试**: 可以通过调用扫描接口，再查询数据库验证指定教练的视频是否已被分类并记录了正确的技术名称和 COS 路径。

**验收场景**:

1. **给定** COS 上存在多个教练课程目录（如小孙、郭焱、高云娇等），**当** 管理员触发全量扫描任务，**那么** 系统将每个视频文件与对应教练姓名、技术类别、COS 路径一起写入数据库，重复扫描不产生重复记录。
2. **给定** 某个视频文件名包含"正手劈长"，**当** 系统解析该文件名，**那么** 该视频的技术类别被记录为 `forehand_push_long`（正手劈长），而不是笼统的 `forehand`。
3. **给定** 某个视频文件名无法匹配任何已知技术关键词，**当** 系统处理该视频，**那么** 该视频被标记为 `unclassified`，并记录原始文件名供人工复核。

---

### 用户故事 2 - 查询指定教练的技术分类列表 (优先级: P2)

研究人员通过 API 查询某位教练（如"孙浩泓"）教授了哪些技术，以及每种技术对应多少个视频，方便规划后续知识库提取的优先级。

**优先级原因**: 查询能力是技术分类数据库的直接价值体现，支撑后续批量提取的任务规划。

**独立测试**: 可以通过 API 请求指定教练名，验证返回的技术列表和视频数量与数据库一致。

**验收场景**:

1. **给定** 数据库已完成扫描，**当** 请求查询"孙浩泓"的技术分类，**那么** 返回该教练所有技术类别及每类视频数量（如：正手攻球 4 条、正手下旋拉球 2 条）。
2. **给定** 查询不存在的教练姓名，**当** 发起请求，**那么** 返回空列表而非错误。

---

### 用户故事 3 - 按技术类别筛选视频并获取 COS 路径 (优先级: P2)

下游任务（后续知识库提取模块）按技术类别批量获取待处理视频的 COS 路径列表，支持按教练、技术类别、是否已处理等条件过滤。

**优先级原因**: 技术分类数据库的最终用途是为知识库提取提供结构化输入，此场景直接支撑后续流程。

**独立测试**: 可以通过按技术类别查询 API，验证返回的 COS 路径在 COS 上真实存在。

**验收场景**:

1. **给定** 数据库已完成扫描，**当** 按技术类别 `forehand_topspin`（正手拉球）过滤，**那么** 返回所有匹配视频的 COS 路径、教练姓名、视频时长（若可获取）。
2. **给定** 指定 `processed=false` 条件，**当** 查询未处理的正手拉球视频，**那么** 只返回尚未提取过知识库的视频。

---

### 用户故事 4 - 增量更新（新视频自动纳入分类）(优先级: P3)

当 COS 上新增视频文件后，管理员可触发增量扫描，只处理数据库中尚未登记的新文件，不重复扫描已有记录。

**优先级原因**: 减少重复工作，为未来持续运营提供支持。

**独立测试**: 在数据库已有记录基础上，手动添加一个模拟新视频记录后触发增量扫描，验证只新增了该条记录。

**验收场景**:

1. **给定** 数据库已有 100 条记录，COS 新增 5 个视频，**当** 触发增量扫描，**那么** 数据库新增 5 条记录，原有 100 条不受影响。

---

### 边界情况

- 当视频文件名为纯数字或英文（无中文技术关键词）时，标记为 `unclassified` 并记录。
- 当同一视频文件名包含多个技术关键词（如"正手劈长与拉球对比"），选择主要技术或拆分为多条记录？[假设: 取第一个匹配的主要技术，同时保留原始文件名]
- 当 COS 请求超时或鉴权失败时，任务标记为失败并记录错误，不影响已成功处理的记录。
- 当目录层级不符合预期格式（如无子目录的平铺文件）时，系统能正常处理并归类。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须遍历 `COS_VIDEO_ALL_COCAH` 路径下所有子目录，通过静态映射配置（目录名 → 教练姓名）识别教练姓名，无匹配时回退使用目录名全称；并枚举每个子目录下的视频文件列表。
- **FR-002**: 系统必须基于视频文件名中的技术关键词，将每个视频分类到精细的乒乓球技术类别（参见技术分类表）。
- **FR-003**: 技术分类必须区分易混淆的相近技术，最低要求覆盖下表中的所有技术类别：

  | 技术类别 ID | 中文名称 | 说明 |
  |-------------|----------|------|
  | forehand_attack | 正手攻球 | 基础正手击打 |
  | forehand_topspin | 正手拉球/上旋拉球 | 正手弧圈（上旋） |
  | forehand_topspin_backspin | 正手拉下旋 | 对下旋球发起正手拉球 |
  | forehand_loop_fast | 正手前冲弧圈 | 速度型弧圈 |
  | forehand_loop_high | 正手高调弧圈 | 旋转型弧圈 |
  | forehand_push_long | 正手劈长 | 正手将球推至对方底线 |
  | forehand_flick | 正手拧拉/台内挑打 | 近台正手反击 |
  | backhand_attack | 反手攻球 | 基础反手击打 |
  | backhand_topspin | 反手拉球/上旋拉球 | 反手弧圈 |
  | backhand_topspin_backspin | 反手拉下旋 | 对下旋球发起反手拉球 |
  | backhand_flick | 反手弹击/快撕 | 近台反手反击 |
  | backhand_push | 反手推挡/搓球 | 反手控制类技术 |
  | serve | 发球 | 各类发球技术 |
  | receive | 接发球 | 接发球技术 |
  | footwork | 步法 | 步伐移动 |
  | forehand_backhand_transition | 正反手转换 | 左右转换衔接 |
  | defense | 防守 | 防弧圈、防快攻等 |
  | penhold_reverse | 直拍横打 | 直拍反手技术 |
  | stance_posture | 站位/姿态 | 基础姿势和站位 |
  | general | 综合/通用 | 无法归入具体技术 |
  | unclassified | 待分类 | 文件名无法识别 |

- **FR-004**: 系统必须将每条分类记录保存以下信息：教练姓名、课程系列名称、COS 完整路径（cos_object_key）、文件名、主技术类别 ID（tech_category）、副技术标签数组（tech_tags，可为空）、原始技术描述（从文件名提取）、分类来源（`rule` 或 `llm`）、分类置信度（0.0-1.0）、视频时长（秒，若可获取）、是否已提取知识库、创建时间、最后更新时间。
- **FR-005**: 系统必须提供 API 接口，支持按教练姓名、技术类别、是否已处理查询分类记录列表。
- **FR-006**: 系统必须提供 API 接口，触发全量扫描和增量扫描任务。
- **FR-007**: 增量扫描必须以 COS 路径（cos_object_key）为唯一标识，跳过已存在记录，不产生重复数据。
- **FR-008**: 系统必须提供按教练统计各技术类别视频数量的汇总接口。
- **FR-009**: 系统必须提供单条记录的技术类别修正接口，允许将分类来源更新为 `manual` 并覆盖原有 `rule`/`llm` 分类结果，同时记录修改时间。

### 关键实体

- **CoachVideoClassification（教练视频技术分类）**: 代表一个已分类的教练教学视频。关键属性：教练姓名、课程系列、COS 路径、文件名、主技术类别（tech_category）、副技术标签数组（tech_tags）、分类来源（rule/llm）、置信度、是否已处理。一个视频对应一条记录。
- **TechCategory（技术类别）**: 乒乓球技术的精细分类枚举，包含技术 ID 和中文名称。作为分类记录的引用标准，不单独持久化。

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 全量扫描完成后，`COS_VIDEO_ALL_COCAH` 下 95% 以上的视频文件均有对应的分类记录（允许 5% 以内的 `unclassified`）。
- **SC-002**: 正手劈长、正手拉下旋、正手前冲弧圈等相近技术的分类准确率（人工抽检 20 条）≥ 90%；其中规则命中记录（source=rule）准确率应 ≥ 95%，LLM 辅助记录（source=llm）准确率应 ≥ 85%。
- **SC-003**: 全量扫描 500 个视频文件的耗时不超过 5 分钟。
- **SC-004**: 增量扫描在 COS 新增视频后能在下一次触发时准确识别并入库，不产生重复记录。
- **SC-005**: 按技术类别查询接口响应时间 ≤ 500ms。

## Clarifications

### Session 2026-04-21

- Q: 一个视频对应一条记录还是多条（多技术）？ → A: 一条记录，支持主技术类别 + 副技术标签数组（tech_tags），主技术为首选分类字段。
- Q: 教练姓名如何从 COS 目录名提取？ → A: 维护静态映射配置（目录名 → 教练姓名），无匹配时回退使用目录名全称作为教练名。
- Q: 是否提供人工修正分类的入口？ → A: 提供 PATCH API 接口更新单条记录的技术类别，来源标记为 `manual`，覆盖原有 `rule`/`llm` 分类。
- Q: 扫描触发方式？ → A: 纯手动，通过 API 触发全量/增量扫描，无定时调度。

## 假设

- COS 目录结构为两级：`COS_VIDEO_ALL_COCAH/{课程系列目录}/{视频文件.mp4}`，教练姓名从课程系列目录名中提取（如"孙浩泓"、"郭焱"）。
- 技术分类采用关键词规则优先策略：先用关键词规则匹配主技术类别，规则未命中时调用 LLM（venus_proxy）辅助分类；每条记录标注分类来源（`rule`/`llm`）和置信度，不涉及视频内容分析（视频内容理解在后续功能中处理）。
- 视频文件均为 `.mp4` 格式。
- 本功能不触发任何知识库提取逻辑，仅完成分类数据库建设。
- 扫描任务通过已有的后台任务队列（Celery）异步执行，API 返回任务 ID 供查询进度；不设定时调度，仅支持手动触发。
- 教练姓名通过静态映射配置文件（`config/coach_directory_map.json`，目录名 → 教练姓名）解析；无匹配时使用完整目录名，并标记为 `name_source=fallback`。
- 视频时长通过读取文件元数据获取，如 COS 未返回则记录为 null。
