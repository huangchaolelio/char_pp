# 研究报告: 非专业选手动作诊断与评分

**功能**: 011-amateur-motion-diagnosis  
**日期**: 2026-04-23  
**状态**: 已完成（回顾性文档）

## 已解决的设计决策

### D-001: 同步 vs 异步处理
- **决策**: 采用同步 POST 端点，请求阻塞直到诊断完成（60秒超时），结果直接返回
- **理由**: MVP阶段应最小化复杂度。异步任务队列（如Celery/RabbitMQ）会引入额外的基础设施、状态管理和轮询逻辑。诊断流程（视频下载+姿态估计+分割+分类+打分+LLM生成）在60秒内通常可完成，同步设计满足MVP需求
- **备选方案**: 
  - 异步任务队列 + 回调通知（实现成本高，需要消息队列、后台worker、客户端轮询或webhook）
  - Webhook回调（要求客户端支持接收回调，增加客户端复杂度）

### D-002: 数据库表设计
- **决策**: 创建独立的两个新表：`diagnosis_reports`（主记录，UUID主键）和`diagnosis_dimension_results`（详细结果，BigSerial主键），与现有的`athlete_motion_analyses`/`deviation_reports`解耦
- **理由**: 
  - 现有表与`AnalysisTask`专家视频分析流程紧耦合，假设有关联的expert task和athlete信息
  - 非专业选手诊断是独立的、无需user认证的流程，不适合复用expert分析表
  - 新表设计保持数据模型清晰，避免在共享表中混合两种不同的分析模式
- **备选方案**: 
  - 扩展现有表（会引入null字段和逻辑分支，降低可维护性）
  - 单一大表（违反规范化，难以查询和维护）

### D-003: 评分算法
- **决策**: 线性插值算法基于标准点的半宽范围（AD-003标准）
  - 计算：`half_width = (max - min) / 2`，`center = (min + max) / 2`
  - 若距离 `d <= half_width` → 得分=100（完美）
  - 若 `half_width < d <= 1.5 * half_width` → 得分线性从100降至60（轻微偏差）
  - 若 `d > 1.5 * half_width` → 得分线性从60降至0（显著偏差）
- **理由**: 三级评价体系与教练反馈直观对应，线性插值提供连续的分数梯度，避免阈值跳跃的生硬感，符合真实运动诊断的连续性特征
- **备选方案**: 
  - 离散三级评价（无中间分值，反馈不够精细）
  - 高斯分布模型（过度复杂，难以解释）

### D-004: LLM 建议生成
- **决策**: 使用LLM动态生成改进建议（调用 `LlmClient.from_settings().chat(messages, temperature=0.3)`），非预定义规则模板
- **理由**: 
  - LLM生成的建议更具自然语言质量，避免模板生成的生硬感，提升用户体验
  - 诊断场景多样化（不同技术、不同问题类型），规则表维护成本高
  - 现有LLM集成已成熟，直接复用降低开发成本
- **容错**: 若LlmError发生，降级返回通用建议模板，不中断诊断流程
- **备选方案**: 
  - 预定义规则表（维护成本高，灵活性差）
  - 纯数据反馈无建议（用户无改进指导）

### D-005: 用户身份与会话
- **决策**: 采用匿名模式，每次诊断请求生成唯一的报告UUID，不绑定用户身份或会话认证
- **理由**: 
  - MVP阶段应聚焦核心诊断功能，用户认证系统是独立的大功能
  - 匿名模式降低准入门槛，支持非注册用户快速体验
  - 外部集成方可独立管理其用户ID与诊断报告UUID的映射
- **跟踪**: 报告UUID足以供后续查询和审计，无需user表绑定
- **备选方案**: 
  - 强制认证（增加MVP复杂度和用户摩擦）
  - 可选认证（额外的条件逻辑，需要处理已认证/未认证的双路径）

### D-006: 视频本地化策略
- **决策**: 支持两种路径
  - COS路径（`cos://bucket/key`）：通过 `cos_client.download_to_temp(key)` 下载到临时文件
  - 本地路径：直接使用（无需下载）
  - 均在finally块中清理临时文件
- **理由**: 
  - COS支持云端存储的视频（如训练平台上传的视频），本地路径支持离线场景
  - 明确的清理策略避免磁盘堆积
  - 两种路径的转换在本地统一接口，对下游的姿态估计器透明
- **备选方案**: 
  - 仅支持COS（离线场景无法处理）
  - 仅支持本地路径（云端存储视频需外部下载，不便集成）

### D-007: 技术类别（Tech Category）验证
- **决策**: 技术类别为必填字段，验证必须匹配 `ActionType` 枚举（当前12个标准值）
- **理由**: 
  - 确保诊断结果与标准知识库关联正确，是后续查询和分析的关键维度
  - 在请求层进行验证，防止无效分类进入数据库
  - 枚举限制保持字段值的一致性和可维护性
- **备选方案**: 
  - 可选技术类别（无法关联标准点，诊断无法定位）
  - 自由文本（无法与标准知识库匹配）

## 依赖研究

### Feature 010 标准知识库接口
- **使用的表**: `tech_standards`、`tech_standard_points`
- **查询条件**: `status='active'` 且 `tech_category` 与请求匹配
- **交互方式**: 读取标准点的 `min_value` / `max_value`，计算评分算法中的阈值范围
- **文档参考**: Feature 010 spec 中的标准点模式（21类技术，每类1-10个标准点）

### 现有 LLM 集成接口
- **初始化**: `LlmClient.from_settings()` 从配置文件读取endpoint、model、key
- **调用**: `.chat(messages, temperature=0.3)` 进行对话式生成
- **错误处理**: 捕获 `LlmError` 异常，降级返回通用建议模板
- **并发模式**: LLM调用在后台线程执行（`asyncio.get_event_loop().run_in_executor(None, ...)`），避免阻塞异步事件循环

### 现有视频处理 Pipeline
诊断流程复用现有的动作分析模块（按顺序调用）：
1. **PoseEstimator**: `pose_estimator.estimate_pose(video_path)` → 逐帧关键点坐标
2. **ActionSegmenter**: `action_segmenter.segment_actions(frame_results)` → 动作片段的时间边界
3. **ActionClassifier**: `action_classifier.classify_segments(segments, target_action)` → 分类置信度
4. **TechExtractor**: `tech_extractor.extract_tech_points(classified, frame_results)` → 技术特征点提取
5. **诊断模块** (新): 通过 `DiagnosisService.diagnose()` 聚合特征点，对照标准点评分，生成诊断报告

### 数据库版本管理
- **Alembic 版本**: 1.13+
- **迁移**: 创建 `diagnosis_reports` 和 `diagnosis_dimension_results` 表的迁移脚本

## 未解决项 / 已知限制

### L-001: 多视频诊断
当前实现仅支持单视频单次诊断。若用户需对同一选手的多个视频进行对比分析（如训练进度追踪），需要在后续版本中：
- 支持批量上传
- 实现诊断结果的时间序列比较
- 添加进度追踪表

### L-002: 自定义标准点范围
诊断算法依赖Feature 010预定义的标准点范围。若特定教练或训练营需自定义范围（如考虑个体差异），当前系统无法支持。后续可考虑：
- 添加可配置的范围调整系数
- 支持教练级别的参数覆盖

### L-003: 多模态诊断（非仅视频）
当前设计专注视频动作分析。其他诊断数据源（如加速度计、压力传感器、运动员自评表）需要独立的集成工作和数据融合策略，未在MVP中包含。
