# 功能规范: 非专业选手动作诊断与评分

**功能分支**: `011-amateur-motion-diagnosis`
**创建时间**: 2026-04-23
**状态**: 草稿
**输入**: 用户描述: "非专业动作诊断,给出评分,总结优点,提供提升建议.可以先使用教练的视频作为测试样本"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 提交视频并获取动作诊断报告 (优先级: P1)

非专业乒乓球爱好者上传一段自己打球的视频，系统与专业教练的动作标准进行比对，
生成一份诊断报告，包含综合评分、优点总结和改进建议。

**优先级原因**: 这是本功能的核心价值交付，其他故事均依赖此基础流程能正常运行。

**独立测试**: 可通过提交一段教练视频（已有标准答案），验证系统输出的评分与预期吻合，
独立验证 MVP 是否可用。

**验收场景**:

1. **给定** 用户提交一段包含可识别乒乓球动作的视频，**当** 系统完成处理，**那么** 返回包含综合评分（0–100）、各维度得分、优点列表和改进建议的诊断报告。
2. **给定** 视频中的动作技术类别可被识别（如正手拉球），**当** 系统匹配对应标准，**那么** 报告中列出该技术类别的标准参考值以及用户动作的偏差量。
3. **给定** 使用教练本人的视频作为输入，**当** 系统进行诊断，**那么** 综合评分应显著高于平均水平（≥ 80 分），验证基准可信度。

---

### 用户故事 2 - 按维度查看详细偏差分析 (优先级: P2)

用户希望了解自己在每个具体动作维度（如手肘角度、挥拍轨迹、重心转移）上与标准的差距，
以便有针对性地练习。

**优先级原因**: 总体评分提供第一印象，维度级别的详情才是用户改进的行动依据。

**独立测试**: 可单独验证某一维度（如 elbow_angle）的偏差计算是否准确，
不依赖其他维度或最终评分逻辑。

**验收场景**:

1. **给定** 系统已完成视频处理，**当** 用户查看诊断报告，**那么** 每个维度显示：用户测量值、标准理想值、标准范围（P25–P75）、偏差程度（正常/轻度偏差/明显偏差）。
2. **给定** 某维度值在标准范围内，**当** 显示该维度结果，**那么** 标记为"达标"并计入优点列表。
3. **给定** 某维度值超出标准范围，**当** 显示该维度结果，**那么** 标记为"需改进"并在建议中提供具体指导。

---

### 用户故事 3 - 使用教练视频作为测试样本验证系统 (优先级: P2)

开发和测试阶段使用已有教练视频（数据库中已有完整 KB 数据）作为输入，
验证诊断系统的准确性和可靠性，为后续面向非专业用户上线做准备。

**优先级原因**: 教练视频提供已知高质量基准，可在没有真实非专业用户的情况下完成端到端验证。

**独立测试**: 从数据库中选取任意已有分类视频，提交诊断，验证报告格式正确且评分合理。

**验收场景**:

1. **给定** 系统中已有教练视频文件路径及其分类信息，**当** 以该视频路径发起诊断请求，**那么** 系统正常返回完整报告，不报错。
2. **给定** 教练视频对应的动作类别已有标准数据（如 forehand_topspin），**当** 完成诊断，**那么** 报告中引用的标准版本与数据库 active 版本一致。

---

### 用户故事 4 - 历史诊断记录查询 (优先级: P3，推迟到后续版本)

> **范围说明（MVP 阶段不实现）**: MVP 阶段采用匿名诊断模式，无用户账户体系，无法跨请求聚合历史。本故事推迟到引入用户身份管理后实现。

**优先级原因**: 对 MVP 不是必须，但对长期用户留存有价值。

---

### 边界情况

- 视频中无法识别出有效乒乓球动作时，系统如何提示用户？
- 对应技术类别尚无标准数据（tech_standards 表中无 active 记录）时如何处理？
- 视频质量过低（模糊、光线不足、遮挡严重）导致关键点提取失败时如何降级处理？
- 同一视频中出现多种技术动作时：用户在提交时必填技术类别，系统仅分析该指定类别的动作片段。
- 非专业用户的动作值极端偏离标准（超出 P25/P75 范围 3 倍以上）时，评分如何计算？

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须接受视频文件或视频存储路径作为诊断请求输入。
- **FR-002**: 诊断请求必须包含用户指定的技术类别参数（如正手拉球、反手推挡），系统仅对该指定类别进行诊断和标准匹配，不自动识别或多类别并行处理。
- **FR-003**: 系统必须从视频中提取与标准对应的动作维度指标（如肘关节角度、挥拍轨迹比、重心转移比、触球时序）。
- **FR-004**: 系统必须将提取的用户指标与对应技术类别的 active 标准（ideal/min/max）进行逐维度比对，计算偏差。
- **FR-005**: 系统必须基于各维度偏差生成综合评分（0–100 分），评分规则须在系统内一致可复现。
- **FR-011**: 诊断接口采用同步处理——单次 POST 请求阻塞直到报告完整生成后返回，不提供异步轮询端点。
- **FR-006**: 诊断报告必须包含：综合评分、各维度测量值与标准对比、优点列表（达标维度）、改进建议列表（偏差维度）。
- **FR-007**: 改进建议必须具体指向偏差维度，说明用户当前值与理想值的差距方向（偏高/偏低）；建议内容由大语言模型根据偏差数据动态生成（复用项目现有 LLM 集成），确保描述自然易懂，用户无需专业知识即可理解。
- **FR-008**: 当对应技术类别无 active 标准时，系统必须返回明确的错误提示，而非生成空报告。
- **FR-009**: 诊断请求和结果必须持久化存储，支持后续历史查询。
- **FR-010**: 系统必须支持使用系统内已有教练视频（通过视频 ID 或存储路径）发起诊断，用于测试和验证。

### 关键实体 *(如果功能涉及数据则包含)*

- **DiagnosisReport（诊断报告）**: 一次诊断的完整结果，包含唯一请求 ID、综合评分、技术类别、所用标准版本、维度详情列表、优点摘要、改进建议列表，关联到输入视频路径；无用户账户关联（MVP 阶段匿名）。
- **DimensionResult（维度结果）**: 单个动作维度的诊断详情，包含维度名称、用户测量值、标准理想值、标准范围、偏差量、达标状态。
- **ImprovementAdvice（改进建议）**: 针对某一偏差维度的文字建议，说明差距方向和改进要点；内容由 LLM 根据维度名称、用户测量值、标准理想值及偏差方向动态生成。
- **TechStandard（技术标准，来自 Feature 010）**: 诊断比对的基准，使用 active 版本的 ideal/min/max 指标。

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 教练视频诊断评分 ≥ 80 分，验证系统基准可信度（使用已有教练视频作为样本）。
- **SC-002**: 诊断报告从视频提交到结果返回的端到端处理不超过 60 秒。
- **SC-003**: 对于已有 active 标准的技术类别，维度达标判断与人工核对的一致率 ≥ 90%。
- **SC-004**: 改进建议覆盖所有偏差维度，无遗漏，且每条建议描述清晰，用户无需专业知识即可理解。
- **SC-005**: 当技术类别无标准数据时，100% 返回明确错误提示，不生成误导性报告。

## Clarifications

### Session 2026-04-23

- Q: 诊断处理模式是同步还是异步？ → A: 同步——POST 请求阻塞直到报告生成，一次性返回完整结果。
- Q: 同一视频中出现多种技术动作时如何处理？ → A: 用户在提交请求时必填技术类别参数，系统仅对该指定类别进行诊断。
- Q: 诊断报告如何标识与用户关联？ → A: 匿名模式——每次诊断生成唯一请求 ID，无用户账户概念；US4 历史查询推迟到后续版本。
- Q: 诊断处理的具体时延目标是多少？ → A: 60 秒内返回完整报告。
- Q: 改进建议内容的来源是规则模板还是 LLM 生成？ → A: LLM 生成——将偏差数据送入大语言模型，生成自然语言改进建议（复用项目现有 LLM 集成）。

## 假设

- 用户上传的视频格式与 Feature 001/002 已支持的格式相同，无需新增视频格式适配。
- Feature 010 构建的技术标准数据库已包含足够的 active 标准，至少覆盖 forehand_topspin 和 backhand_push 两个主要类别，可支持 MVP 验证。
- 视频中的动作技术类别识别复用 Feature 004（视频技术分类）的现有能力，本功能不重新实现分类逻辑。
- 动作维度指标提取复用 Feature 001/002 已有的姿态分析管线（MediaPipe + 关键点计算），本功能不重新实现底层提取。
- 综合评分采用各维度得分的加权平均，权重在初始版本中各维度等权，后续可调整。
- 维度偏差判断规则：值在 [min, max] 范围内为"达标"，超出范围但在 1.5 倍范围内为"轻度偏差"，超出 1.5 倍范围为"明显偏差"。
- MVP 阶段采用匿名模式：每次诊断生成唯一请求 ID，无用户账户概念，不支持跨请求历史聚合（US4 推迟到后续版本）。
- 测试阶段优先使用数据库中已有的教练视频（通过 COS 路径或 task_id 引用），无需用户上传真实非专业视频。
