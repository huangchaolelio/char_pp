# 功能规范: 优化视频提取知识库的处理耗时

**功能分支**: `007-processing-speed-optimization`
**创建时间**: 2026-04-21
**状态**: 草稿
**输入**: 用户描述: "优化视频提取知识库的处理耗时"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 上传视频后更快获得知识库结果 (优先级: P1)

管理员上传专家教练视频后，系统在更短的时间内完成视频拆分、姿态估计与知识库提炼，使管理员可以更快地审阅和使用提炼出的技术要点与教学建议。

**优先级原因**: 处理耗时直接影响管理员的工作效率。当前 5–10 分钟视频的处理耗时接近视频时长本身，导致等待感强烈；缩短耗时是用户最直接的痛点。

**独立测试**: 提交一个 5 分钟的教练视频，记录从提交到任务状态变为 `completed` 的总耗时，与优化前基线对比，验证耗时下降幅度。

**验收场景**:

1. **给定** 一个时长约 5 分钟的专家教练视频，**当** 管理员提交处理任务后，**那么** 任务在 100 秒内完成（当前基线约 216 秒，目标缩短 ≥ 50%）
2. **给定** 一个时长约 10 分钟的专家教练视频，**当** 提交任务后，**那么** 任务在 200 秒内完成（当前基线约 310 秒，目标缩短 ≥ 35%）
3. **给定** 优化前已成功处理的视频，**当** 用优化后系统处理相同视频时，**那么** 提炼结果（技术要点数量、参数值）与优化前完全一致，无精度损失

---

### 用户故事 2 - 批量处理多个视频时系统资源利用率提升 (优先级: P2)

管理员批量提交多位教练的视频时，系统能够并发利用服务器的多核 CPU 资源，减少总体等待时间，而不是串行逐段处理每一个任务。

**优先级原因**: 服务器已有 Tesla T4 GPU 用于姿态估计，但视频预分割（CPU 密集型）仍为串行瓶颈，CPU 核心利用率低；改善并发可直接降低批量处理的总耗时。

**独立测试**: 同时提交两个视频任务，观察两个任务的处理是否真正并行推进（任务日志时间戳交叉），总耗时不超过单任务的 1.5 倍。

**验收场景**:

1. **给定** 同时有两个视频任务在队列中，**当** 系统开始处理时，**那么** 两个任务的视频预分割阶段并发执行，CPU 使用率可见提升
2. **给定** 一个包含 8 个片段的视频，**当** 进行预分割时，**那么** 各片段并行生成，预分割总耗时相比串行降低 ≥ 60%

---

### 用户故事 3 - 系统稳定性在优化后保持不变 (优先级: P1)

优化后，现有的所有测试全部通过，任务失败率不增加，处理结果的正确性与优化前保持一致。

**优先级原因**: 性能优化若引入不稳定性（竞争条件、资源泄漏、结果错误），代价远超收益；稳定性是优化的前提条件。

**独立测试**: 运行全量回归测试套件，验证 0 失败；对 3 个不同长度的视频做端到端处理，对比优化前后的提炼结果。

**验收场景**:

1. **给定** 优化后的系统代码，**当** 运行全量测试套件时，**那么** 全部测试通过，与优化前测试数量一致
2. **给定** 一个历史成功处理的视频，**当** 用优化后系统重新处理时，**那么** 提炼出的技术要点数量和核心参数值偏差在 ±5% 以内（允许浮点计算微小差异）
3. **给定** 并发处理两个视频，**当** 两个任务同时进行预分割时，**那么** 两个任务均成功完成，无一失败

---

### 边界情况

- 当视频片段数量超过 4（最大并发上限）时，超出的片段排队等待，不会同时启动超过 4 个 ffmpeg 进程
- 当 `/tmp` 磁盘空间不足时，并行写入多个临时文件会加速触发磁盘满错误；系统应在预分割前检查可用空间
- 当视频编码格式不支持 `-c copy` 直接拷贝时，系统自动回退到重编码模式，不中断任务
- 当某一片段 ffmpeg 命令失败时，立即取消其余所有进行中的并行片段，整体任务标记为 `failed`，需管理员重新提交
- GPU 显存有限（Tesla T4 16GB），姿态估计保持原有串行逐片段方式，不并行，避免显存溢出

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须将视频预分割阶段改为并行处理，最大并发数为 4，在多核 CPU 上同时生成各视频片段
- **FR-002**: 系统必须将视频片段编码模式从重编码改为直接流拷贝（避免不必要的视频重编码），对支持此模式的视频格式生效
- **FR-003**: 并行预分割的最大并发数固定上限为 4；实际并发数取 `min(4, 片段总数)`
- **FR-004**: 当直接流拷贝不可用时，系统必须自动回退到原有重编码模式，不中断任务
- **FR-005**: 优化后的系统必须与现有 API 接口、任务状态完全兼容，无任何破坏性变更
- **FR-006**: 系统必须同时通过两种方式记录各优化阶段耗时：① worker 日志（格式：`[timing] phase=<阶段名> duration=<秒数>s`）；② 持久化到数据库 `analysis_tasks` 表的 `timing_stats` JSON 字段（记录 `pre_split`、`pose_estimation`、`kb_extraction` 各阶段耗时）
- **FR-007**: 当某一并行片段失败时，系统必须立即取消其余所有进行中的片段，整体任务标记为 `failed`

### 关键实体 *(如果功能涉及数据则包含)*

- **视频片段（Segment）**: 原始视频按时间切割的子片段，是并行处理的最小单位；数量由视频时长和分段策略决定
- **处理任务（AnalysisTask）**: 封装整个视频处理流程（预分割 → 姿态估计 → 知识库提炼），新增 `timing_stats` JSON 字段记录各阶段耗时（秒）

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 5 分钟教练视频的端到端处理耗时从约 216 秒降至 100 秒以内（缩短 ≥ 50%）
- **SC-002**: 10 分钟教练视频的端到端处理耗时从约 310 秒降至 200 秒以内（缩短 ≥ 35%）
- **SC-003**: 视频预分割阶段耗时（多片段）相比串行基线降低 ≥ 60%
- **SC-004**: 优化后全量回归测试通过率 100%，与优化前一致
- **SC-005**: 优化后提炼结果的技术要点数量与优化前一致，核心参数值偏差 ≤ 5%
- **SC-006**: 并发处理 2 个视频任务时，总耗时不超过串行处理的 1.5 倍
- **SC-007**: 每个成功完成的任务，数据库中的 `timing_stats` 字段包含 `pre_split`、`pose_estimation`、`kb_extraction` 三个阶段的耗时数据

## 假设

- 服务器当前已有 Tesla T4 GPU（16GB 显存），姿态估计已使用 GPU 加速（YOLOv8），本次优化不改变姿态估计逻辑
- 优化重点为 CPU 密集型的视频预分割阶段，GPU 已是当前瓶颈中的较小部分
- 服务器 CPU 核心数 ≥ 4，支持 4 路并行预分割
- 视频文件格式为 MP4/H.264，主流格式支持直接流拷贝
- 现有 Celery worker 为单进程，并行化在 worker 内部通过 ProcessPoolExecutor 实现，不依赖增加 Celery worker 数量
- 磁盘空间管理策略不变（临时文件处理完毕后即清理）
- 不引入新的外部依赖，仅使用 Python 标准库（`concurrent.futures`）
- `analysis_tasks` 表新增 `timing_stats` JSON 字段需要对应的数据库迁移
