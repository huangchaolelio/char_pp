# 功能规范: 任务处理管道重新设计与提交限流

**功能分支**: `013-task-pipeline-redesign`
**创建时间**: 2026-04-24
**状态**: 草稿
**输入**: 用户描述: "数据重新初始化,历史数据可以清理掉.重新设计提交任务处理流程和逻辑,不同任务不受影响.每个任务只做一件时,比如视频分类任务只分类,知识库提取任务只提取,诊断任务只诊断.不相互耦合.同时为任务处理性能最大化,限制提交数量.可单个提交,也可批量提交,但要看已提交数量."

## Clarifications

### Session 2026-04-24

- Q: 视频分类任务（video_classification）的执行单位是什么？ → A: API 层只暴露「单条分类」任务；COS 全量/增量扫描作为内部调度任务（定时/运维触发），不走公共任务通道、不占 `video_classification` 通道容量配额。
- Q: 批量请求条数超过 BATCH_MAX_SIZE（默认 100）时如何响应？ → A: 整体拒绝请求（返回 `400 BATCH_TOO_LARGE`，不处理任何条目）；由客户端自行分批重试。
- Q: 分类任务与知识库提取任务的触发关系？ → A: 严格前置。提交 `kb_extraction` 任务时必须检查该视频在 `coach_video_classifications` 中已有有效的 `tech_category` 分类记录；若无，直接返回 `400 CLASSIFICATION_REQUIRED`。两类任务物理解耦，不存在隐式触发。
- Q: 历史任务类型（`expert_video` / `athlete_video`）如何处理？ → A: 彻底删除。数据重置时一并清理所有旧类型任务记录；task_type 枚举不保留旧值，迁移中直接 DROP；API 只接受新三类（`video_classification` / `kb_extraction` / `athlete_diagnosis`）。
- Q: 同一视频重复提交同类型任务时的幂等判断？ → A: 拒绝已完成或进行中。`(cos_object_key, task_type)` 维度：存在 pending/processing/completed 记录则返回 `409 DUPLICATE_TASK`；failed/cancelled 状态允许重新提交；completed 状态重跑需在请求体带 `force=true` 显式覆盖。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 提交单条任务并立刻得到处理 (优先级: P1)

作为知识库管理员，我在系统空闲或繁忙时提交一条「知识库提取」任务，该任务应被立刻分配给对应类型的处理通道，不会被其他类型的任务（例如正在进行的视频分类批量扫描）阻塞。

**优先级原因**：这是当前系统最大的痛点 — 用户刚提交一条任务长时间不被处理，是本次重构的核心触发点。必须最先解决。

**独立测试**：在系统已有 100 条视频分类任务堆积时，单独提交一条知识库提取任务，可观测该任务立即被 `kb_extraction` 通道受理且在 5 秒内进入 `processing` 状态。

**验收场景**：

1. **给定** 分类通道已满负荷（队列长度达上限），**当** 用户单条提交一条知识库提取任务，**那么** 该任务立即进入知识库提取通道，不受分类通道状态影响
2. **给定** 知识库提取通道空闲，**当** 用户提交一条诊断任务，**那么** 诊断任务进入诊断通道，与知识库提取通道互不干扰
3. **给定** 所有通道均空闲，**当** 用户单条提交任务，**那么** 任务从 `pending` 转为 `processing` 的等待时间 ≤5 秒

---

### 用户故事 2 - 批量提交任务时受单一通道限流保护 (优先级: P1)

作为批量数据处理人员，我希望一次性提交数百条同类任务时，系统能根据当前通道已提交（pending + processing）数量限制接受条数，避免队列无限膨胀耗尽资源，同时对已被其他类型任务占用的通道不产生影响。

**优先级原因**：批量提交场景是本项目核心工作流（如全量教练视频分类、全量知识库提取）。缺少限流会导致 Redis 队列堆积、内存溢出、任务相互等待，是引起本次重构的关键需求。

**独立测试**：向空系统批量提交 200 条分类任务，当超过分类通道上限（例如 50）时，剩余请求被拒绝并返回明确错误，其他类型通道不受影响仍可接受任务。

**验收场景**：

1. **给定** 分类通道上限为 N，当前待处理数已达 N，**当** 客户端再提交一条分类任务，**那么** 系统返回 `429 QUEUE_FULL` 错误并告知当前通道容量与占用
2. **给定** 分类通道已满（达到上限），**当** 用户提交知识库提取任务，**那么** 该任务被正常受理（不同通道独立计数）
3. **给定** 客户端批量提交 M 条任务（M > 通道剩余容量），**当** 提交请求到达，**那么** 系统按单条接受规则处理：前 K 条受理（K=剩余容量），后 M-K 条每条返回 `429`，响应明确告知接受/拒绝分布
4. **给定** 单次批量请求，**当** 请求本体携带条数超过批量上限（例如 100），**那么** 系统拒绝整个请求并返回 `400 BATCH_TOO_LARGE`

---

### 用户故事 3 - 每类任务只做自己的事 (优先级: P1)

作为系统维护者，我希望「视频分类」任务只做目录扫描与技术分类，「知识库提取」任务只做音视频解析与知识点入库，「诊断」任务只做偏差分析，三类任务在代码路径、执行资源、队列、Worker 进程上完全解耦，一类任务的崩溃或超时不影响其他两类。

**优先级原因**：当前 `expert_video_task` 耦合了分类推断、知识提取、动作识别多个职责，导致一条慢任务阻塞整个通道；本次重构要让职责单一化，从架构上杜绝耦合。

**独立测试**：分别关闭任一类型 Worker，其他两类任务仍可正常提交、处理并完成，互不感知。

**验收场景**：

1. **给定** 分类 Worker 停止运行，**当** 用户提交知识库提取或诊断任务，**那么** 任务可被正常处理，不会出现等待或错误
2. **给定** 知识库提取任务执行过程中崩溃，**当** 分类任务在同一时刻提交，**那么** 分类任务正常进入自己的通道处理不受影响
3. **给定** 提交一条任务时，**当** 系统接收请求，**那么** 请求参数只需包含该任务类型所需字段，不应携带其他任务类型的参数

---

### 用户故事 4 - 数据重新初始化 (优先级: P2)

作为项目负责人，我希望在启用新任务管道前清空所有历史任务相关数据（任务记录、知识库草稿、诊断结果等非核心资产），保留教练目录扫描出的分类元数据，让新流程从干净状态启动，便于验证新架构效果。

**优先级原因**：历史 2589 条任务记录（大部分 failed 或已废弃）会对新管道的监控指标造成干扰；清理后可直接观测新架构的真实表现。

**独立测试**：执行清理操作后，任务列表为空，教练目录（`coach_video_classifications`）保留完整。

**验收场景**：

1. **给定** 数据库存在历史任务记录，**当** 执行数据初始化，**那么** `analysis_tasks` 及其级联子表被清空，教练与视频目录分类数据保留
2. **给定** 初始化已完成，**当** 提交新任务，**那么** 任务 ID 从空表开始分配，监控接口显示准确计数

---

### 用户故事 5 - 并发能力最大化 (优先级: P2)

作为运营人员，我希望系统在硬件允许范围内最大化吞吐：同类型任务内部并行执行，不同类型任务可同时运行、互不抢夺资源通道。

**优先级原因**：本项目有 1015 条视频需处理，单通道串行已成瓶颈；希望不同类型任务并行、同类型任务内部按资源上限并发。

**独立测试**：同时提交 10 条不同类型任务，观测三类通道均有任务在 `processing` 状态，且各自通道内按配置并发上限运行。

**验收场景**：

1. **给定** 每类通道配置了并发上限（例如分类 1，知识库 2，诊断 2），**当** 提交远超并发数的任务，**那么** 每类通道同时处理的任务数严格等于其并发上限
2. **给定** 三类任务同时被提交，**当** 系统处于运行状态，**那么** 三类通道的 processing 数之和 ≤ 总配置上限，且每类 ≤ 自身上限

---

### 边界情况

- 批量提交 100 条任务时，如通道只剩 10 个槽位，前 10 条接受、后 90 条拒绝：响应应明确列出每条的结果（task_id 或错误原因）
- 一个任务因 Worker 崩溃卡在 `processing` 超过硬超时时间：系统应能自动将其置为 `failed`，释放通道槽位
- 提交时指定了错误的任务类型（不存在的枚举值）：返回 `400`，不能误落到其他通道
- 初始化操作被误触发：应要求显式确认（例如必须传入 confirmation token），防止误删
- 当 Redis 或数据库短暂不可用时，提交接口应快速失败并返回明确错误，不让客户端长时间挂起
- 同一条视频被重复提交同类型任务：应检测并拒绝（或返回已存在的 task_id），不浪费槽位
- 通道并发数配置被改小：已在 processing 的任务允许跑完，不会被强制中止

## 需求 *(必填)*

### 功能需求

#### 任务类型解耦

- **FR-001**: 系统必须将现有任务拆分为三个独立的任务类型：`video_classification`（视频分类）、`kb_extraction`（知识库提取）、`athlete_diagnosis`（运动员诊断），每类任务使用独立的队列、独立的 Worker 进程、独立的数据库状态字段
- **FR-002**: 每类任务的提交接口必须只接受该类任务的必要参数，不得互相掺杂字段（例如诊断任务不再需要传教练视频相关字段）
- **FR-003**: 一类任务的 Worker 停止、崩溃或队列堆积，不得阻塞其他类型任务的接收和处理
- **FR-004**: 每类任务的业务逻辑模块（Celery Task、Service 层）必须物理分离到不同文件，不得共享可变状态
- **FR-004a**: 提交 `kb_extraction` 任务时系统必须校验目标视频在 `coach_video_classifications` 表中已有有效的 `tech_category`；若不存在，返回 `400 CLASSIFICATION_REQUIRED` 并告知需先完成分类；知识库提取任务不得隐式触发分类

#### 提交限流

- **FR-005**: 系统必须为每类任务通道独立维护「当前已提交数」计数（等于 pending + processing 数），且提供查询接口
- **FR-006**: 每类任务通道必须配置「最大积压上限」参数（queue_capacity），当前已提交数 ≥ 上限时拒绝新提交请求，返回 `429 QUEUE_FULL` 及当前容量信息
- **FR-007**: 单次批量提交请求必须支持 N 条任务（N ≤ 批量上限 BATCH_MAX_SIZE，默认 100），请求中的每条任务独立走限流检查，响应必须返回每条的受理结果（接受或拒绝原因）
- **FR-008**: 单条提交必须复用批量提交同一套限流逻辑（相当于 N=1 的批量），保证单/批量行为一致
- **FR-009**: 当通道有空闲槽位时，新提交的任务必须能在 5 秒内被 Worker 拾取开始处理
- **FR-010**: 限流计数必须以数据库任务状态为权威数据源（非 Redis 队列长度），防止数据库与 Redis 不一致时误判

#### 并发与资源管理

- **FR-011**: 每类任务通道必须配置独立的「并发执行数」参数（concurrency），Worker 同时处理的任务数不得超过该值
- **FR-012**: 系统必须支持不同类型任务同时并发执行，CPU/GPU/内存资源按任务类型分组隔离（诊断和知识库提取均需 GPU，可共享资源池但不共享队列槽位）
- **FR-013**: 任务超过硬超时阈值（默认 420 秒）仍未完成时，系统必须自动将其标记为 `failed` 并释放通道槽位
- **FR-014**: Worker 崩溃重启后，之前处于 `processing` 状态但 Redis 队列中无对应消息的任务，必须在启动时被检测并自动标记为 `failed`（即 orphan task 回收）

#### 数据初始化

- **FR-015**: 系统必须提供数据重置能力，能够清空全部历史任务及其级联数据（analysis_tasks / audio_transcripts / coaching_advice / teaching_tips / expert_tech_points / tech_semantic_segments / athlete_motion_analyses / tech_knowledge_bases 的 draft 版本）
- **FR-016**: 数据重置必须保留核心资产数据：教练目录（coaches）、视频分类元数据（coach_video_classifications / video_classifications）、已发布的知识库版本（非 draft 的 tech_knowledge_bases）、标准动作模板（standards）
- **FR-017**: 数据重置操作必须要求显式的确认标识（例如配置项或操作命令的安全参数），不能通过普通 API 误触发

#### 监控与查询

- **FR-018**: 系统必须提供通道状态查询接口，展示每类任务通道的：当前 pending 数、processing 数、容量上限、剩余槽位、最近完成速率
- **FR-019**: 单任务状态查询接口必须按任务类型返回该类型特有的字段（例如诊断任务返回偏差分数，知识库任务返回提取段数）
- **FR-020**: 任务列表查询必须支持按任务类型过滤，分页参数沿用现有约定（page + page_size，默认 20 上限 100）

### 关键实体 *(如果功能涉及数据则包含)*

- **任务通道 (TaskChannel)**：每类任务一个逻辑通道，属性包括通道标识（类型枚举）、最大积压上限、并发执行数、当前 pending 数、当前 processing 数、状态（启用/停用）
- **分类任务 (ClassificationTask)**：只对**单条**视频做技术分类归档（关键字段：COS 对象键、教练 ID、归档的 tech_category、置信度）。注：COS 全量/增量目录扫描不是公共任务通道任务，由运维侧触发的内部调度任务承担，不占用 `video_classification` 通道配额。
- **知识库提取任务 (KbExtractionTask)**：只负责从单条教练视频中提取知识点。关键字段：COS 对象键、教练 ID、语言、提取段数、产生的 KB 草稿版本
- **诊断任务 (DiagnosisTask)**：只负责运动员视频的偏差分析。关键字段：运动员视频键、对比的标准版本、诊断偏差结果
- **提交结果 (SubmissionResult)**：单次提交（单/批量）的响应实体，属性包括：接受总数、拒绝总数、每条请求的结果（task_id 或拒绝错误码）

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 提交一条任意类型的任务，在目标通道有空闲槽位时，从提交 HTTP 响应到 Worker 开始处理的等待时间 P95 ≤ 5 秒
- **SC-002**: 当一类任务通道已满时，另两类任务通道的提交与处理完全不受影响：从接收到开始处理的等待时间不劣化超过 10%
- **SC-003**: 批量提交 100 条同类型任务，系统能在 2 秒内返回响应，响应明确包含每条的受理结果
- **SC-004**: 通道容量上限参数可配置，配置变更后新提交请求在 30 秒内生效
- **SC-005**: 数据初始化操作执行后，任务相关表完全清空，核心资产表数据行数与操作前一致，整个清理在 1 分钟内完成
- **SC-006**: 三类 Worker 独立运行一周后，其中一类因异常崩溃重启，其他两类的任务成功率不下降超过 1%
- **SC-007**: 同等硬件条件下，本方案的三类任务并发总吞吐量相比旧单队列方案提升 ≥ 50%（按每分钟完成任务数计）
- **SC-008**: 卡住的任务（超过硬超时未完成）能在超时后 30 秒内被自动标记失败，通道槽位得到释放

## 假设

- 清理历史任务数据时，用户明确同意丢弃现有 2589 条 analysis_tasks 记录及其 draft 版本的知识库数据；教练分类元数据（coach_video_classifications）及已发布知识库必须保留
- 三类任务默认的通道容量上限假设为：分类 5、知识库提取 50、诊断 20（可通过配置文件调整）；批量单次提交上限默认 100 条
- 并发执行数默认：分类 1（COS 扫描本身是 IO 密集）、知识库提取 2（GPU Whisper + 姿态估计）、诊断 2（GPU 姿态估计）；可通过启动参数调整
- 现有 Celery + Redis 技术栈保留，不引入新的消息中间件
- 存在一个运营侧的「确认令牌」机制可用于安全执行数据初始化，无需引入复杂权限系统
- 当前 `expert_video_task` 中"自动推断动作类型"、"懒加载视频分类"等副作用，将在重构后剥离到分类任务的职责里，知识库提取任务不再隐式触发分类
- 数据库表结构可以按需新增字段（如 task_type 扩展枚举值）；历史任务类型 `expert_video` / `athlete_video` 在数据重置时一并删除，枚举值不保留
- 单条视频重复提交同类型任务的幂等粒度：`(cos_object_key, task_type)` 二元组；存在 pending/processing/completed 记录则拒绝（返回 `409 DUPLICATE_TASK`）；failed/cancelled 可直接重新提交；completed 状态重跑须带 `force=true` 显式覆盖
