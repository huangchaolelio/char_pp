# 功能规范: 知识库提取流水线化（有向图 + 并行）

**功能分支**: `014-kb-extraction-pipeline`
**创建时间**: 2026-04-24
**状态**: 草稿
**输入**: 用户描述: "把kb_extraction 拆分成任务,有依赖关系的放在一个流水线里,无依赖的可并行.原任务重缺少从视频直接提取专业知识库的任务,也增加上（旧版本是有的)"

## 背景

Feature-013 将任务管道物理解耦为三类通道（分类 / KB 提取 / 诊断），但 `kb_extraction` 目前只是最小化存根——只翻转 `kb_extracted=True` 标志，**不再从视频提取任何真实知识**。旧版流水线（Feature-002 的 11 步单体任务）虽然能从视频提取姿态动作点 + 音频讲解要点 + 合并后入库，但耦合在单个 Celery 任务里，任一步失败整条重跑；也无法让互不依赖的子步骤并行。

本 Feature 要把「一次 KB 提取」重新拆成**多个可独立调度的子任务**，按**依赖关系**组成流水线（DAG），无依赖分支并行跑；并补齐 Feature-013 遗漏的「视频直接提取专业知识库」能力（对应旧流水线里的视觉动作要点 + 音频讲解要点 + 合并落库三段）。

## Clarifications

### Session 2026-04-24

- Q: 单次 KB 提取作业的最长允许时长（作业级超时）？ → A: 作业整体 45 分钟超时、单子任务 10 分钟超时；超时后整作业判定为 failed
- Q: 冲突标注（视觉路 vs 音频路参数不一致）的处理回路？ → A: 冲突项单独存放「待审核冲突表」不入正式 KB；两路非冲突条目各自落库；审核后冲突决议补入 KB
- Q: 成功作业能否重跑？ → A: 默认拒绝重复提交（返回既有作业 ID）；接受 `force=true` 强制新建作业覆盖旧结果
- Q: 子任务失败的自动重试策略？ → A: I/O 类子任务（视频下载、音频转写 / LLM 调用）自动重试 2 次 × 30 秒间隔；CPU 密集型子任务（姿态分析、视觉提炼）不自动重试，首次失败即判 failed
- Q: 中间结果（姿态关键点、音频转写）的保留期？ → A: 作业 success 后中间结果保留 24 小时自动清理；failed 状态保留 7 天供排障；之后统一清理
- Q: 视频下载是否作为独立共享子任务？ → A: 是。视频下载为独立子任务，其输出（本地视频文件句柄）被姿态分析、音频转写两路并行消费；同一作业内**只下载一次**
- Q: 作业的并行子任务如何占用 `kb_extraction` 通道的 `concurrency`？ → A: 通道 `concurrency=N` 指同时运行中的作业数 ≤N；每个作业内的并行子任务共享该作业名额（通过异步/线程池内部并发）实现，不额外扣减通道预算
- Q: 作业状态的对外 API 形态？ → A: 纯拉取式 HTTP 查询（`GET /extraction-jobs/{job_id}`），客户端自行轮询；与 Feature-013 `GET /tasks/{task_id}` 保持一致；不引入 SSE/WebSocket/回调
- Q: Worker 崩溃时正在运行的子任务如何恢复？ → A: 崩溃的子任务自动判 failed，下游按规则 skipped，整作业进入 failed 状态，**不自动重跑**；运维通过 `POST /extraction-jobs/{job_id}/rerun` 显式重跑；与 Feature-013 orphan sweep 语义对齐
- Q: 管理员能否列表/分页查看历史作业？ → A: 本 Feature 提供最小分页列表接口 `GET /extraction-jobs?page=X&page_size=Y&status=Z`，支持按状态过滤；不做复杂搜索/排序/多字段过滤（与 Feature-013 分页风格一致）
- Q: 使用 `force=true` 覆盖已成功作业时，旧的冲突表记录如何处理？ → A: 旧冲突项标记为 `superseded_by_job_id` 保留在冲突表但默认审核队列隐藏，新作业冲突项照常入表；旧记录可通过历史查询查阅

## 用户场景与测试 *(必填)*

### 用户故事 1 - 一次 KB 提取自动分解为多阶段子任务（优先级: P1）🎯 MVP

管理员对一个已分类的教练视频提交一次「KB 提取」请求。系统自动把这次提取拆解为多个子任务（视频下载、姿态分析、音频转写、视觉知识点提炼、音频知识点提炼、冲突合并落库等），各子任务按依赖关系组成一条流水线：后置任务等前置任务完成才能开始，彼此无依赖的任务**同时**进行。管理员可以查看这次提取的整体进度（已完成子任务数 / 总子任务数），任何子任务失败时能看到是哪一步失败、失败原因。

**优先级原因**: 这是本 Feature 的核心价值——从「整体跑 / 整体败」的粗粒度任务升级为「子任务级可观测 + 局部重跑」的流水线，直接提升运维效率和失败容错能力。没有这一条，其它用户故事都无法独立演示。

**独立测试**: 提交一次 KB 提取请求，查询流水线状态，验证返回的子任务列表与依赖图；观察耗时显著短于同等子任务串行总和（说明并行生效）。

**验收场景**:

1. **给定** 一个已分类的教练视频，管理员提交一次 KB 提取请求，**当** 请求被接受，**那么** 系统返回一个「提取作业 ID」，并在后续状态查询中返回该作业下所有子任务的清单、依赖关系、各自状态（pending / running / success / failed / skipped）。
2. **给定** 一个运行中的提取作业，音频转写子任务与姿态分析子任务都仅依赖「视频下载」完成，**当** 视频下载成功后，**那么** 这两个子任务**同时**开始运行，不强制串行。
3. **给定** 一个提取作业中途发生失败（例如音频转写超时），**当** 管理员查询状态，**那么** 系统清晰标注哪一个子任务失败、错误原因、哪些后置任务因此被跳过、哪些不依赖它的子任务依然正常完成。
4. **给定** 一次提取作业所有子任务均成功，**当** 合并子任务完成入库，**那么** 作业整体状态为 success，且 `coach_video_classifications.kb_extracted=true`（与现有语义兼容）。

### 用户故事 2 - 补齐「视频 → 专业知识库」能力（优先级: P1）

管理员提交一个已分类的教练视频进行 KB 提取后，系统从**视频画面本身**（姿态关键点序列）和**音频讲解**（教练口头说的技术要点）两路独立提取专业知识条目，最终合并入库。每条知识条目带来源标注（"视觉" / "语音"），参数冲突时标注冲突而非静默覆盖。没有这能力之前，Feature-013 的 `kb_extraction` 只是占位符，不生成任何真实知识。

**优先级原因**: 旧版 11 步流水线已经验证过「视觉+音频双路提取」的价值（Feature-002 已上线）。Feature-013 拆解管道时把业务逻辑剥离成 stub，留下空洞。补齐这能力是让 `kb_extraction` 真正产生业务价值的必要条件。

**独立测试**: 提交一个有清晰讲解音频的教练教学视频，验证最终知识库条目既包含姿态维度（如"拉球时肘部角度 90°-120°"），也包含来源为"语音"的条目（如"重心要前移"）；若两路给出的参数有冲突，作业结果中明确列出冲突项。

**验收场景**:

1. **给定** 一段含教练讲解语音的教学视频，**当** 提取作业成功完成，**那么** 入库的知识条目中至少包含 1 条来源标注为「语音」的条目，且至少包含 1 条来源标注为「视觉」的条目。
2. **给定** 教练在视频中口述"正手拉球肘部角度 90°-120°"，**当** 提取作业完成，**那么** 知识库中该维度的取值范围与讲解一致，±5° 偏差内。
3. **给定** 视觉路与音频路对同一维度的参数不一致（如视觉测得 105° 对口述 130°），**当** 合并子任务执行，**那么** 该维度的两路参数被写入独立的「待审核冲突表」等待人工审核，**不**进正式 KB；视觉路和音频路各自不冲突的条目照常入 KB。
4. **给定** 视频中无可识别音频（全静音或非支持语言），**当** 作业执行，**那么** 音频路子任务返回「音频不可用」而不失败，视觉路正常产出，合并子任务只合入视觉路的知识条目。

### 用户故事 3 - 子任务并行带来显著速度提升（优先级: P1）

对同一个已分类视频发起 KB 提取，并行后的总耗时应显著短于同等工作量的串行流水线。管理员不需要为了拿更快的结果而手动拆分视频或提交多次请求。

**优先级原因**: 这是能用数字直接衡量的业务价值（P1 即速度）。用户描述里明确要求"无依赖的可并行"。

**独立测试**: 取一段 10 分钟带音频讲解的教练视频，分别以「强制串行模式」（全部子任务依次跑）和「默认并行模式」运行同一个作业，对比两者总耗时，并行至少快 30%。

**验收场景**:

1. **给定** 一个作业中「姿态分析」与「音频转写」无互相依赖，**当** 作业在默认模式下运行，**那么** 两个子任务的时间区间在监控上重叠（说明并行），整条作业总耗时不等于两者之和。
2. **给定** 「视觉知识点提炼」依赖「姿态分析」，「音频知识点提炼」依赖「音频转写」，**当** 前置任务各自完成，**那么** 这两个提炼子任务**立即且并行**开始（不等对方）。
3. **给定** 合并落库子任务同时依赖两路提炼完成，**当** 两路都成功，**那么** 合并任务才开始；若一路失败另一路成功，合并任务在降级模式下只处理成功的那一路。

### 用户故事 4 - 局部失败可重跑、不拖垮整条流水线（优先级: P2）

单个子任务失败后，管理员可以重跑该子任务（及其下游依赖），**不需要**重新下载视频、重做已成功的前置步骤。并行的其他分支即使其中一路彻底失败，另一路已完成的成果不丢失。

**优先级原因**: 这是并行流水线相对于单体任务的主要运维优势。P2 因为即使没这能力，用户也能用 US1 的重新提交来覆盖，只是代价更高。

**独立测试**: 提交一个作业，人工在数据库或 Redis 注入一个"音频转写失败"状态，然后触发重跑单步；验证重跑只消耗该子任务及其下游的资源，视频下载和姿态分析不重复执行。

**验收场景**:

1. **给定** 一个作业中「音频转写」子任务失败、其余子任务已成功，**当** 管理员发起「重跑失败子任务」请求，**那么** 系统只重跑该子任务及下游「音频知识点提炼」和「合并落库」，不重跑视频下载和姿态分析。
2. **给定** 重跑请求已处理完成，**当** 查询作业状态，**那么** 整条作业状态从 failed 变为 success，且已成功的前置子任务输出（如姿态关键点）被直接复用，无二次计算。
3. **给定** 并行两路中一路失败、一路成功的作业，**当** 查询状态，**那么** 成功那一路的输出保留可查（如视觉提取的条目），不因另一路失败被回滚。

---

### 用户故事 5 - 与 Feature-013 通道/限流体系无缝衔接（优先级: P2）

`kb_extraction` 通道原本的容量（50）和并发（2）是对「整个作业」计数，不是对「作业中每一个子任务」计数——否则一个作业就会占满整条通道。子任务的调度不能绕过通道限流，但也不能因为按子任务计数导致通道规模需要翻 N 倍。

**优先级原因**: 避免破坏 Feature-013 已上线的限流/容量治理能力。P2 因为这是兼容性约束而非直接业务价值。

**独立测试**: 同时提交 50 个 KB 提取作业（达到通道容量上限），验证通道显示 50 个 pending 作业而不是 50 × 子任务数个 pending；验证在途作业的子任务不额外占用通道容量。

**验收场景**:

1. **给定** `kb_extraction` 通道容量 50，**当** 提交第 51 个作业，**那么** 系统返回 `QUEUE_FULL`，与 Feature-013 保持一致。
2. **给定** 通道内已有 1 个运行中的作业（内部由 5 个子任务组成），**当** 查询通道状态，**那么** 处理中数为 1，不是 5。
3. **给定** 一个运行中的作业因子任务失败最终判定为 failed，**当** 管理员发起重跑，**那么** 重跑不消耗新的通道容量（同一个作业 ID 重启），除非作业已被清理归档。

### 边界情况

- 视频下载子任务失败（例如 COS 对象被删除）→ 整条作业早期失败，下游子任务自动标记 skipped，不启动，避免浪费算力
- 音频语言不受支持或视频无音频 → 音频路子任务标记 `skipped` 或返回空，不算失败；视觉路不受影响
- 并行子任务同时对同一个 `coach_video_classifications` 行写中间结果 → 每个子任务写独立的中间态字段（不互相踩）；只有合并子任务写最终 `kb_extracted`
- 作业被管理员中途取消 → 已排队未启动的子任务撤销；运行中子任务收到取消信号并尽快退出，已产生的中间结果保留供后续 debug
- 重跑失败子任务时，下游任务此前已经是 success → 下游必须一起重跑（因为上游结果变了），不允许部分复用
- 超过保留期后对 failed 作业发起重跑 → 中间结果已清理，必须从视频下载整条链路重走；系统在查询接口中明确提示「中间结果已过期，重跑将从头开始」
- 视频过长（远超旧上限 90 分钟）→ 由上游提交层或分段子任务分片处理，不让单个子任务超时，该阈值沿用旧版 90 分钟

## 需求 *(必填)*

### 功能需求

#### 流水线骨架与编排

- **FR-001**: 每一次 KB 提取请求必须创建一条「作业」记录，作业下包含一组按依赖图组织的子任务；作业整体状态由所有子任务的完成情况推导
- **FR-002**: 子任务必须具备明确的依赖声明；调度器在所有依赖子任务成功后才能启动该子任务，无依赖或依赖已完成的子任务**必须并行**调度
- **FR-003**: 作业状态查询接口（拉取式 HTTP，与 Feature-013 `GET /tasks/{task_id}` 风格一致；不提供 SSE/WebSocket/回调）必须返回子任务清单、各自状态（pending/running/success/failed/skipped）、开始/结束时间、失败原因；客户端通过轮询获取实时状态
- **FR-004**: 单个子任务失败时，其下游依赖子任务必须标记为 skipped，不得继续执行；并行的独立分支不受影响继续运行
- **FR-005**: 管理员必须能够对「已失败的作业」发起重跑，重跑只重新执行失败子任务与其下游依赖子任务，已成功的前置子任务输出直接复用

#### 核心子任务能力（对应旧版 11 步流水线中的业务子步骤）

- **FR-006**: 必须提供「视频下载」子任务：从 COS 下载视频到 Worker 本地临时目录；同一作业内只下载一次；其输出（本地视频文件句柄）作为共享中间结果被姿态分析和音频转写两路子任务**并行消费**，不允许为每路分别重复下载；下载失败时下游均标记 skipped
- **FR-007**: 必须提供「姿态分析」子任务：对本地视频提取骨骼关键点序列，保存为中间结果；输入依赖视频下载
- **FR-008**: 必须提供「音频转写」子任务：对本地视频提取音频轨道并转写为带时间戳的文本；输入依赖视频下载
- **FR-009**: 必须提供「视觉知识点提炼」子任务：从姿态关键点序列提取技术要点（如关节角度范围、重心偏移），输出带来源标注「视觉」的知识条目；输入依赖姿态分析
- **FR-010**: 必须提供「音频知识点提炼」子任务：从音频转写文本识别教练口头陈述的技术要点，输出带来源标注「语音」的知识条目；输入依赖音频转写
- **FR-011**: 必须提供「合并落库」子任务：同时依赖视觉、音频两路提炼的输出；两路对同一技术维度给出一致参数时合并后直接入正式 KB；两路参数冲突时，冲突项写入独立的「待审核冲突表」（**不**进正式 KB），两路各自的非冲突条目仍正常入 KB；仅此子任务有权翻转 `coach_video_classifications.kb_extracted=true`

#### 执行语义

- **FR-012**: 视觉路与音频路必须可以完全独立地失败或成功——若音频路失败，视觉路的结果仍可经合并子任务（在降级模式下）入库，不因音频失败而丢弃
- **FR-013**: 所有中间结果（姿态关键点、音频转写）必须以结构化形式保存在作业生命周期内可访问的位置，重跑时可直接复用，不需要二次计算；保留期：作业最终状态为 success 后保留 24 小时自动清理，failed 状态保留 7 天后清理；超过保留期后重跑需重新执行前置子任务
- **FR-014**: 作业级别的整体耗时必须低于同等工作量的串行执行耗时（具体指标见 SC-003）

#### 与 Feature-013 的兼容

- **FR-015**: 一个作业对 `kb_extraction` 通道的占用必须计为 1，不得按子任务数扩大占用；通道 `concurrency=N` 指同时运行中的作业数 ≤ N；一个作业内的并行子任务共享该作业的通道名额（通过异步/线程池内部并发），不再额外扣减通道预算；通道容量语义与 Feature-013 保持一致
- **FR-016**: 重跑一个失败作业不得消耗新的通道容量
- **FR-017**: 作业必须沿用 Feature-013 的幂等键规则（相同 `cos_object_key` 在同一时间窗内只有一个有效作业）；已成功的作业默认拒绝重复提交，直接返回既有作业 ID；接受 `force=true` 参数时例外——允许强制新建作业并覆盖旧结果（旧作业标记为 archived 保留供审计；旧作业关联的「待审核冲突项」打上 `superseded_by_job_id` 标记，默认审核队列隐藏但历史查询可见，保留审计链）

#### 可观测性

- **FR-018**: 作业状态查询接口必须能在 1 秒内返回子任务执行时间线摘要
- **FR-019**: 所有子任务失败必须记录明确的失败原因（不得只有「unknown error」）；错误信息在查询接口中直接暴露，管理员无需翻日志
- **FR-020**: 必须支持作业级和子任务级两层超时：作业整体超时 45 分钟，单子任务超时 10 分钟；任一层超时即标记该作业/子任务 failed，并触发下游 skipped 传播
- **FR-021**: I/O 类子任务（视频下载、音频转写、LLM 调用等依赖外部服务的步骤）必须支持自动重试，上限 2 次，首次重试间隔 30 秒；CPU 密集型子任务（姿态分析、视觉/音频知识点提炼）不自动重试，首次失败即判 failed；自动重试次数不计入作业级超时扣减之外的额外预算（重试发生在子任务 10 分钟超时窗口内）
- **FR-022**: Worker 崩溃导致子任务处于孤儿状态时（超过合理探测阈值仍为 running），系统必须自动将该子任务判为 failed，下游按依赖传播标记 skipped，整作业进入 failed 状态；**不自动重跑**，等待管理员通过显式重跑接口触发；与 Feature-013 的 `celeryd_after_setup` sweep 语义保持一致
- **FR-023**: 必须提供作业列表分页查询接口 `GET /extraction-jobs`：支持 `page`、`page_size`（默认 20、上限 100，与 Feature-013 一致）、`status` 过滤参数；返回每条作业的 job_id / 提交时间 / 整体状态 / 对应视频 key；不做复杂搜索/多字段过滤/排序（若需要，通过 Feature-013 `GET /api/v1/tasks` 按 task_type 检索）

### 关键实体

- **KB 提取作业（Extraction Job）**: 一次 KB 提取的顶级容器；关键属性包括作业 ID、对应的教练视频、创建时间、整体状态、整体耗时；与「子任务」一对多
- **子任务（Pipeline Step）**: 流水线中的单个可执行步骤；关键属性包括步骤类型（下载/姿态/音频/视觉提炼/音频提炼/合并）、所属作业 ID、上游依赖步骤、当前状态、开始/结束时间、失败原因、输出句柄
- **中间结果（Intermediate Artifact）**: 子任务输出的数据（关键点序列、转写文本、知识条目草稿等），供下游子任务消费；生命周期随作业
- **冲突标注（Conflict Flag）**: 合并子任务在两路参数冲突时产出的标注记录；存在独立的「待审核冲突表」中；包含维度名、视觉值、音频值、作业 ID、创建时间、`superseded_by_job_id`（可空，用于 `force` 覆盖时隐藏旧记录）；等待人工审核，审核后将结果补入正式 KB；不阻塞入库主流程

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 任意一次 KB 提取作业都能返回完整子任务清单与依赖关系视图，查询延迟 ≤1 秒
- **SC-002**: 当音频路与视觉路子任务均可独立执行时，两者时间区间重叠（并行），作业整体耗时比同等串行执行至少节省 30%
- **SC-003**: 一个典型 10 分钟教学视频的 KB 提取作业整体耗时不超过旧版 Feature-002 单体任务耗时的 90%（证明并行化没有引入显著调度开销）
- **SC-004**: 任意单个子任务失败后，管理员可通过状态查询明确知道哪一步失败、失败原因、哪些下游被跳过；90% 的运维人员在 1 分钟内能据此判断下一步操作（重跑 / 放弃 / 联系开发）
- **SC-005**: 重跑一个失败子任务的资源消耗（时间 + 算力）不高于该子任务首次执行成本的 110%（即复用已有中间结果、不重做前置）
- **SC-006**: 作业级别对 `kb_extraction` 通道的占用始终为 1，子任务数增加不导致通道显示的在途作业数放大
- **SC-007**: 视觉+音频双路合并的知识库条目，与旧版 Feature-002 对相同视频的输出在知识条目数上差异 ≤20%（证明业务能力补齐成功，不是退化版）

## 假设

- 单次作业典型子任务数量为 6（下载 / 姿态 / 音频转写 / 视觉提炼 / 音频提炼 / 合并），未来扩展空间较大；不硬编码「6 步」
- `kb_extraction` 通道的现有容量（50）和并发（2）针对「作业级别」计数，不随子任务数缩放
- 视频最大处理时长沿用旧版 90 分钟上限；更长视频需要上游分片（本 Feature 不处理）
- 中间结果的临时存储假设用 Worker 本地文件系统 + 数据库结构化字段；不引入新的外部存储组件；具体存储介质划分（哪些字段用 DB，哪些用本地 FS）交由 `/speckit.plan` 阶段决定。保留期：success 作业 24 小时、failed 作业 7 天（由后台清理任务执行）
- 新增子任务类型不影响 Feature-013 已定义的三类 `task_type` 枚举——本 Feature 的「子任务」在作业内部调度，不占据 `analysis_tasks.task_type` 枚举值（保持 Feature-013 的三值枚举稳定）
- 冲突标注不阻塞入库主流程——冲突项进入独立的「待审核冲突表」等待人工审核，非自动解决；人工审核 UI/API 不在本 Feature 范围，仅提供存储层供未来对接
- 重跑逻辑假设作业和子任务状态均持久化；临时存活的 Celery 元数据不足以支撑，需自己的存储层
- 旧版 Feature-002 的知识点提炼逻辑（视觉规则 + LLM 抽取）原则上可直接复用，不在本规范中重新设计算法——本 Feature 的重点在编排与拆解，而不在单子任务的算法改进
