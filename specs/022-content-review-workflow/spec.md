## 背景与变更说明 *(修正版必读)*

本次修正在 v1（初版）基础上扩大功能范围，将原来"在 TRAINING 阶段内部插入 content_review 步骤"升级为**业务流程顶层重构**：

- **v1 范围（已废弃）**：把 `curate_segments` 与 `extract_kb` 拆开，并在两者之间插入 `content_review`，整体仍归属 TRAINING 阶段
- **v2 范围（本规范）**：把"素材入库到审核通过"全流程拆为独立的**内容准备（CONTENT_PREP）**阶段，TRAINING 阶段从此仅消费"已审核通过"的内容做知识库提取。整体业务流程由当前的三阶段升级为四阶段：
  - **CONTENT_PREP（内容准备）**：素材归集 → 视频预处理 → 技术分类 → 内容清洗 → **内容审核**
  - **TRAINING（训练）**：知识库提取 → 冲突审阅 → KB 版本激活
  - **STANDARDIZATION（建立标准）**：技术标准构建（保持不变）
  - **INFERENCE（诊断）**：学员诊断（保持不变）

> 测试阶段说明（章程原则 XI）：项目当前处于测试阶段，未声明上线。本规范不为存量数据迁移做兼容设计；存量数据如需补登 `CONTENT_PREP` 阶段标记，作为独立完善任务在功能 MVP 交付后处理。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 业务流程升级为四阶段 (优先级: P1)

平台运营负责人希望把当前的"训练-建标-诊断"三阶段升级为"内容准备-训练-建立标准-诊断"四阶段，让"原始素材进入到形成可用语料"这一段长流程从训练阶段中独立出来，单独度量、单独排队、单独失败重试，避免清洗/审核异常拖累 KB 抽取的产能账本。

**优先级原因**: 这是本次修正的根本目的；阶段拆分一旦完成，运营、SRE、PM 就能在"内容准备"维度独立观测吞吐与质量，TRAINING 阶段的成功率统计才不会被前置链路噪声污染。

**独立测试**: 调用任一对外暴露的阶段视图接口（或查询任务列表中的 `phase` 字段），可以看到 `CONTENT_PREP` / `TRAINING` / `STANDARDIZATION` / `INFERENCE` 四个独立阶段的任务计数与状态分布；新提交的素材任务被归入 `CONTENT_PREP`，而 KB 抽取任务被归入 `TRAINING`。

**验收场景**:

1. **给定** 一段新上传到 COS 根路径的教练视频，**当** 系统识别并入库后，**那么** 该条目的"所属阶段"为 `CONTENT_PREP`，且不会出现在 TRAINING 阶段的待办列表里
2. **给定** 一条已经被审核通过的视频条目，**当** 系统派发 KB 抽取任务时，**那么** 该任务的"所属阶段"为 `TRAINING`，且其前置链路（素材归集/预处理/分类/清洗/审核）都标记为已完成
3. **给定** 运营查看阶段级监控视图，**当** 选择"内容准备"阶段时，**那么** 应能看到独立于 TRAINING 阶段的吞吐、积压、失败率指标，互不混算
4. **给定** STANDARDIZATION 与 INFERENCE 阶段的现有功能，**当** 流程升级为四阶段后，**那么** 这两个阶段的对外行为、API 路径、状态机判据保持不变（只调整其文档位置与上游依赖来源）

---

### 用户故事 2 - 审核门作为内容准备阶段的最终判据 (优先级: P1)

运营审核员希望在内容清洗完成后，能在审核台上对每条候选语料做"通过/拒绝"决策；只有通过审核的内容才会进入训练阶段，未通过或未审核的内容绝不会被 KB 抽取意外消费。

**优先级原因**: 这是把审核作为"阶段出口门禁"而非"中间步骤"的关键；它直接决定了 TRAINING 阶段的输入纯度，是知识库质量的根本保障。

**独立测试**: 在审核台上对一条已清洗的视频做"通过"操作，立刻提交一次 KB 抽取，能成功入队；对另一条做"拒绝"或保留"未审核"状态，提交 KB 抽取应被拒绝并返回明确错误。

**验收场景**:

1. **给定** 一条视频已完成内容清洗，**当** 内容准备阶段判定其状态时，**那么** 状态为"待审核"，且阶段不会自动推进到 TRAINING
2. **给定** 一条视频处于"待审核"状态，**当** 审核员标记其为"通过"，**那么** 该条目的内容准备阶段被判定为完成，可被 TRAINING 阶段的 KB 抽取消费
3. **给定** 一条视频处于"待审核"或"审核拒绝"状态，**当** 任何调用方尝试为其触发 KB 抽取，**那么** 系统必须以明确的错误码拒绝，并提示"内容审核未通过/未完成"
4. **给定** 一条已被审核通过的视频，**当** 其源素材被重新清洗（例如规则更新后重跑清洗），**那么** 之前的审核结论自动失效，需重新审核才能再次进入 TRAINING

---

### 用户故事 3 - 审核工作台与统计 (优先级: P2)

运营审核员希望有一个集中的审核工作台，按教练/技术类别筛选待审核条目，查看清洗后的关键摘要（片段时长、抽取出的关键帧/字幕示例、规则命中信息），快速做出审核判断，并能随时回看自己的审核历史与团队的审核统计。

**优先级原因**: 工作台体验直接决定审核吞吐；没有集中入口，审核就会成为整个内容准备阶段的瓶颈。

**独立测试**: 审核员登录后能在一个列表里看到所有"待我审核"的条目；选中一条能看到清洗摘要并提交决策；提交后该条从待审核列表中消失，进入"已审核"列表。

**验收场景**:

1. **给定** 系统中有多条"待审核"视频，**当** 审核员打开审核工作台，**那么** 能看到列表（含教练、技术类别、清洗时间、片段数量），并支持按教练/技术类别筛选
2. **给定** 审核员选择一条待审核条目，**当** 查看详情，**那么** 能看到清洗后的关键摘要（片段总数、保留率、若干样例片段的起止时间与转写片段）
3. **给定** 审核员提交"通过"或"拒绝（含理由）"决策，**当** 操作成功后，**那么** 该条目从"待审核"列表移除，决策连同审核员、时间被持久记录
4. **给定** 运营负责人查看审核统计，**当** 选择某时间窗，**那么** 能看到该窗内的审核总量、通过率、平均审核时延、人均吞吐

---

### 用户故事 4 - 阶段级可观测性与回滚 (优先级: P3)

SRE 希望"内容准备"作为一个独立阶段，拥有自己的指标维度（阶段进入/退出计数、阶段平均停留时长、按步骤的失败分布），并在审核门出现规则误伤导致 TRAINING 输入枯竭时，能够通过运营级开关临时绕过审核门做应急回滚。

**优先级原因**: 阶段化重构的可运维性是长期收益的关键；没有阶段级指标和回滚剧本，重构反而会增加 SRE 的认知负担。

**独立测试**: 查看监控仪表盘可以看到 `phase=CONTENT_PREP` 的独立曲线；在演练环境切换审核门为"绕过"后，待审核条目可被 TRAINING 直接消费，演练完成后切回"严格"恢复正常。

**验收场景**:

1. **给定** 监控系统已接入新阶段维度，**当** 任一任务被记录时，**那么** 其指标 tag 中的 `phase` 取值为四阶段之一，方便分阶段聚合
2. **给定** 因清洗规则误伤导致 TRAINING 输入断流，**当** 运营触发"审核门绕过"开关并登记审计，**那么** 已清洗未审核的条目可被 TRAINING 直接消费，且回滚动作在审计日志中可追溯
3. **给定** 应急绕过结束，**当** 运营切回"严格审核门"，**那么** 系统恢复"必须审核通过才能进入 TRAINING"的默认行为，且不会遗留任何永久跳过的旧条目

---

### 边界情况

- **重复提交审核**：同一视频清洗结果被重复提交进入审核台时，系统以最新一次清洗结果为准，旧的待审核记录自动作废
- **清洗失败 vs 审核拒绝**：清洗失败的视频不进入审核台（停在内容准备阶段的清洗步骤），审核拒绝的视频进入"已归档拒绝"分组，两者在统计中分别记账
- **审核员变更/账号停用**：某审核员账号停用后，其名下的"待审核"条目应可重新分配，不出现孤儿条目
- **极长积压**：当待审核积压超过运营定义的红线（例如 24 小时未处理），系统应在阶段级监控上突出告警，但不阻塞清洗与素材归集步骤继续运行
- **跨阶段错放**：任何尝试把 `CONTENT_PREP` 阶段的步骤手工提交到 TRAINING 队列、或反之的请求，必须被拒绝并返回明确错误，避免阶段越界
- **审核结论失效后的下游**：当某视频的审核结论被失效（如内容重洗），如果其下游已有 KB 抽取任务在排队/进行，系统应允许该已派发任务继续完成（不级联中止），但新提交的抽取必须被拒绝直至重新审核

## 需求 *(必填)*

### 功能需求

#### 阶段重构类
- **FR-001**: 系统必须在业务阶段维度上把现有 TRAINING 阶段拆分为两个独立阶段：`CONTENT_PREP`（内容准备）与 `TRAINING`（训练），与现有的 `STANDARDIZATION`、`INFERENCE` 一起构成四阶段
- **FR-002**: 系统必须把"素材归集 / 视频预处理 / 技术分类 / 内容清洗 / 内容审核"五个步骤归属到 `CONTENT_PREP` 阶段
- **FR-003**: 系统必须把"知识库提取 / 冲突审阅 / KB 版本激活"三个步骤归属到 `TRAINING` 阶段（其语义、状态机、API 路径保持向后一致）
- **FR-004**: 系统必须保证 `STANDARDIZATION`（技术标准构建）与 `INFERENCE`（学员诊断）两个阶段的对外行为、API 路径、状态机判据**不因本次重构发生破坏性变化**
- **FR-005**: 系统必须为每个面向任务/条目的查询接口增加"所属阶段"字段，取值为四阶段之一，便于按阶段聚合与筛选

#### 审核门类
- **FR-006**: 系统必须在内容清洗完成后，把视频条目自动置为"待审核"，且不会自动推进到 TRAINING 阶段
- **FR-007**: 系统必须提供审核工作台，至少支持"待审核列表 / 详情查看 / 通过 / 拒绝（含理由）/ 已审核历史"五项基础能力；审核粒度为**整段视频条目**（与 `coach_video_classifications` 行一一对应），不做片段级拆分（澄清 Q1）
- **FR-007a**: 审核员角色采用最简模型——**单一 `reviewer` 角色**，admin 视为 reviewer 的超集；任意 reviewer 均可对任意"待审核"条目下决策，最后一次写入为准；首版**不**支持申诉、二审、分级、分组、审计员等扩展角色（澄清 Q2）
- **FR-008**: 系统必须把"审核通过"作为 `CONTENT_PREP` 阶段的完成判据；只有通过该判据的视频条目才能被 TRAINING 阶段的 KB 抽取消费
- **FR-009**: 系统必须在 KB 抽取入口处校验来源条目的审核状态；当审核状态为"未审核 / 已拒绝 / 已失效"时，必须以明确错误码拒绝抽取请求，错误码集中登记于 `src/api/errors.py`，建议命名：`CONTENT_NOT_REVIEWED` / `CONTENT_REVIEW_REJECTED` / `CONTENT_REVIEW_STALE`
- **FR-010**: 系统必须持久化每一次审核决策，至少包含：`reviewer_id`（审核员主键）、`decision`（通过/拒绝）、`reason_code` + `note`（拒绝必填，通过可选）、`decided_at`（决策时间）、所审核的 `cleansing_version`（清洗版本标识）（澄清 Q2/Q5）
- **FR-010a**: 拒绝条目必须**保留 DB 行**并将状态置为 `rejected`，**不**软删除、**不**物理删除；COS 物理文件**不同步删除**；审核工作台默认列表自动过滤 `rejected`，但支持通过显式查询参数（`?state=rejected`）单独查看（澄清 Q5）
- **FR-011**: 当一条视频被重新清洗（清洗版本递进）时，系统必须使其先前的审核结论自动失效（状态置为 `stale`），需要重新审核
- **FR-011a**: 清洗版本失效**不**级联中止已入队 / running 的 `extract_kb` 任务（这些任务正常跑完并落库），但**新提交**的 KB 抽取在入口校验阶段被 `CONTENT_REVIEW_STALE` 拒绝（澄清 Q3）

#### 可观测与回滚类
- **FR-012**: 系统必须为新增的 `CONTENT_PREP` 阶段提供独立的阶段级监控锚点，至少包含"进入/退出计数、阶段停留时长、各步骤失败分布"
- **FR-013**: 系统必须为审核步骤提供步骤级监控锚点，至少包含"待审核积压量、通过率、平均审核时延、人均吞吐"
- **FR-014**: 系统必须提供运营级"审核门绕过"开关，开关切换需登记审计；切换为"绕过"后，TRAINING 阶段允许消费"已清洗未审核"的条目；切回"严格"后立即恢复默认行为且不留遗留豁免

#### 统计与配置类
- **FR-015**: 系统必须提供按时间窗的审核统计能力（总量 / 通过率 / 平均时延 / 人均吞吐）
- **FR-016**: 系统必须支持"待审核积压红线阈值"的配置化（例如 24 小时），积压超过红线时在阶段级监控中以告警形式突出，但不阻塞内容准备阶段其他步骤继续运行

#### 性能与索引类
- **FR-017**: 审核工作台列表接口必须在**典型在用规模（50–200 条/日新增、累计 ≤ 50 万条）**下，对常用筛选组合（`coach_id` / `tech_category` / `review_state` / 时间窗）做服务端筛选与分页；目标性能：`page_size ≤ 50` 时 **P95 < 500 ms**，`page_size = 100` 时 P95 < 1 s（澄清 Q4）
- **FR-018**: `coach_video_classifications` 表必须新增覆盖常用审核查询的复合索引，建议至少包含：
  - `(review_state, decided_at DESC)` — 驱动"待审核列表 / 已审核历史"主路径
  - `(coach_id, review_state)` — 驱动按教练过滤的列表
  - `(tech_category, review_state)` — 驱动按技术类别过滤的列表

### 关键实体

- **业务阶段（Phase）**: 四阶段枚举之一（`CONTENT_PREP` / `TRAINING` / `STANDARDIZATION` / `INFERENCE`），每条任务/作业实体必须显式归属其中之一
- **内容审核条目（Content Review Item）**: 一条状态为 `pending_review` / `approved` / `rejected` / `stale` 的审核条目，**承载在既有 `coach_video_classifications` 表上（不另建子粒度表）**，关联一个清洗版本与最近一次决策；粒度 = 整段视频条目（澄清 Q1）
- **审核决策（Review Decision）**: 单次审核动作的留痕（`reviewer_id` / `decision` / `reason_code` / `note` / `decided_at` / `cleansing_version`）；拒绝条目永久保留，COS 文件不删除（澄清 Q5）
- **审核员（Reviewer）**: 单一角色 `reviewer`（admin 为其超集），由 `reviewer_id` 主键标识，无分级/分组/二审（澄清 Q2）
- **审核门开关（Review Gate Switch）**: 全局开关，取值"严格 / 绕过"，绑定审计日志
- **阶段级度量锚点（Phase Observability Anchor）**: 阶段进入/退出计数、阶段平均停留时长、阶段内步骤失败分布的统一记录位

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 业务阶段重构上线后，运营在阶段级仪表盘上 100% 能区分"内容准备 / 训练 / 建立标准 / 诊断"四阶段的吞吐与积压，互不混算

- **SC-007**: "审核门绕过"开关从切换到生效的时延 ≤ 30 秒，且任意一次切换 100% 留下审计记录

- 假设运营团队具备审核标准（内容质量 / 技术相关性 / 教练权威性 / 视频质量）并能稳定执行；本规范不定义具体审核打分细则，仅承载"通过/拒绝"二元决策
