# 功能规范: 常用 SQL 查询脚本整理

**功能分支**: `009-sql-query-scripts`
**创建时间**: 2026-04-22
**状态**: 草稿
**输入**: 整理常用查询 SQL 脚本，按分类组织，方便排查定位和使用

## 用户场景与测试 *(必填)*

### 用户故事 1 - 查看任务处理进度 (优先级: P1)

开发者或运维人员需要快速了解当前视频分析任务的整体状态：哪些在处理中、哪些卡死、哪些失败。

**优先级原因**: 批量处理时最高频使用，直接影响问题定位效率。

**独立测试**: 直接执行查询，返回各状态任务数及占比。

**验收场景**:

1. **给定** 数据库中存在多种状态的任务，**当** 执行进度查询 **那么** 返回各状态计数及占比
2. **给定** 存在超过 N 分钟无进展的 processing 任务，**当** 执行僵尸任务查询 **那么** 返回卡死任务列表

---

### 用户故事 2 - 按教练查看视频到知识库全链路 (优先级: P1)

需要追踪某位教练的视频从分类、任务提交、转录、知识库提取的完整状态。

**优先级原因**: 排查某教练数据缺失或质量问题时的核心路径。

**独立测试**: 传入教练姓名，返回该教练的全链路数据状态。

**验收场景**:

1. **给定** 教练姓名，**当** 查询该教练视频 **那么** 返回视频列表、kb_extracted 状态、关联 task 状态
2. **给定** 教练姓名，**当** 查询 teaching_tips **那么** 返回该教练所有已提取的建议

---

### 用户故事 3 - 按技术类别查看知识库内容 (优先级: P1)

按乒乓球技术类别（如 forehand_topspin）查看提取进度、教学建议、技术要点。

**优先级原因**: 知识库质量评估和技术分析的核心查询。

**独立测试**: 传入技术类别 ID，返回该类别完整知识库状况。

**验收场景**:

1. **给定** 技术类别，**当** 查询提取进度 **那么** 返回已提取/待提取/失败的视频数
2. **给定** 技术类别，**当** 查询 teaching_tips **那么** 按教练分组返回所有建议

---

### 用户故事 4 - 排查失败和超时任务 (优先级: P2)

定位失败、rejected 或超时 (processing 状态但长时间无进展) 的任务，获取错误信息。

**独立测试**: 执行查询后可直接获得需要重试的任务列表及错误原因。

**验收场景**:

1. **给定** 存在 failed/rejected 任务，**当** 执行失败查询 **那么** 返回任务 ID、视频路径、错误信息
2. **给定** 指定技术类别，**当** 查询未提取视频 **那么** 返回 kb_extracted=false 的完整列表

---

### 边界情况

- 某教练有视频但没有对应的 analysis_task（尚未提交）
- 某视频有多个 analysis_task（重试导致）
- teaching_tips 置信度过低的记录如何过滤
- processing 任务的时间判断（created_at 还是 started_at）

## 需求 *(必填)*

### 功能需求

- **FR-001**: 脚本文件按查询类别组织为独立 `.sql` 文件，存放于 `specs/009-sql-query-scripts/queries/` 目录
- **FR-002**: 每个 SQL 文件顶部必须有注释说明用途、参数、返回字段
- **FR-003**: 涉及参数替换的查询，用 `:param_name` 或 `-- REPLACE: xxx` 注释标注可替换位置
- **FR-004**: 查询结果列名须使用中文别名，方便非技术人员阅读
- **FR-005**: 目录下须提供 `README.md` 作为查询索引，列出所有脚本及用途

### 查询类别覆盖范围

**类别 A: 任务进度与状态**
- A1: 全局任务状态汇总（各状态计数+占比）
- A2: 僵尸任务检查（processing 超过 N 分钟无进展）
- A3: 失败/rejected 任务列表（含错误信息）
- A4: 最近 N 小时任务处理速率

**类别 B: 视频分类统计**
- B1: 所有技术类别视频数统计
- B2: 各教练视频数及 kb_extracted 进度
- B3: 待提取视频列表（按技术类别筛选）
- B4: 指定技术类别下各教练分布

**类别 C: 按教练查链路**
- C1: 指定教练全部视频及任务状态
- C2: 指定教练的 teaching_tips（按技术类别分组）
- C3: 指定教练视频→任务→转录→知识库完整状态

**类别 D: 按技术类别查知识库**
- D1: 指定技术类别提取进度汇总
- D2: 指定技术类别所有 teaching_tips（按教练分组）
- D3: 指定技术类别的 expert_tech_points（技术参数维度）
- D4: 指定技术类别各教练建议数量对比

**类别 E: 音频转录**
- E1: 转录质量分布（ok/low_snr/silent 等）
- E2: 指定任务的转录内容（句子数、总时长）
- E3: 无转录但任务成功的视频（缺失排查）

**类别 F: 知识库版本管理**
- F1: 所有知识库版本状态
- F2: 指定版本覆盖的动作类型及技术要点数

### 关键实体

- **coach_video_classifications**: 视频分类记录，`cos_object_key` 为唯一标识
- **analysis_tasks**: 视频分析任务，通过 `video_storage_uri = cos_object_key` 关联分类
- **teaching_tips**: 教学建议，通过 `task_id → analysis_tasks → coach_video_classifications` 三级 JOIN 关联
- **expert_tech_points**: 技术参数要点，不可变设计，通过版本号管理
- **audio_transcripts**: 音频转录，`sentences` 为 JSONB 数组
- **tech_knowledge_bases**: 知识库版本元数据

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 覆盖至少 6 个类别、20 条以上常用查询，满足日常排查 90% 场景
- **SC-002**: 每个查询有清晰注释，非技术人员可在 5 分钟内理解用途和使用方式
- **SC-003**: 所有查询在测试数据库中可直接执行，无语法错误
- **SC-004**: README 索引清晰，可在 1 分钟内定位到目标查询

## 假设

- 数据库连接信息从 `.env` 读取，脚本中不内嵌连接字符串
- 执行环境为 PostgreSQL 14+，语法兼容
- `video_storage_uri` 与 `cos_object_key` 值相同（可直接 JOIN）
- teaching_tips 的高质量过滤默认使用 `confidence >= 0.9`
- 僵尸任务判断：`started_at < NOW() - INTERVAL '30 minutes'` 且 `status = 'processing'`
