# 功能规范: Skill KB 到参考视频

**功能分支**: `003-skill-kb-to-reference-video`
**创建时间**: 2026-04-20
**状态**: 草稿
**输入**: 将"专业教学视频 → 专业知识库"提炼流程封装为可重复执行的 Skill，完成提炼后自动生成标准参考视频供管理员审核。

---

## 背景

Feature-001 实现了单次上传专家视频后提取技术知识库（KB）的流程，Feature-002 增加了音频增强提取能力。但每次提炼都需要管理员手动上传视频、填写参数、逐步操作；当需要定期更新 KB（如每月追加新教学视频）时，重复操作成本高、易出错。

Feature-003 将这一提炼流程封装为可配置、可重复执行的 **Skill**：管理员只需配置一次 Skill（指定视频来源、动作类型等参数），后续每次点击"执行"即可触发完整的 KB 提炼流程，并在提炼完成后自动生成一段**标准参考视频**（按知识库中各技术维度剪辑、标注、拼接），供管理员预览后一键审批入库或驳回重做。

---

## 用户场景与测试

### 用户故事 1 - Skill 配置与执行 (优先级: P1)

管理员在系统中创建一个"正手攻球标准提炼"Skill，配置好 COS 视频前缀或指定已完成分析的任务 ID 列表，点击"执行"后系统自动完成：视频批量分析 → 技术要点汇总 → KB 草稿生成 → 参考视频生成，全程异步，管理员可随时查看进度。

**优先级原因**: 这是整个 Feature 的核心价值——将手动多步操作变为一键触发的可复现流程，大幅降低 KB 维护成本。

**独立测试**: 创建一个包含 2 个已分析任务的 Skill，触发执行，验证产出的 KB 草稿包含来自两个视频的技术要点，且参考视频已生成。

**验收场景**:

1. **给定** 管理员创建了包含 3 个视频任务的 Skill，**当** 点击执行，**那么** 系统返回 execution_id，执行状态为 `pending`，并在后台异步处理。
2. **给定** Skill 执行成功，**当** 查询执行状态，**那么** 状态为 `success`，`kb_version` 字段包含新生成的 KB 版本号。
3. **给定** Skill 的视频来源为空列表，**当** 尝试触发执行，**那么** 系统返回 422 错误，error.code 为 `SKILL_NO_VIDEOS`。

---

### 用户故事 2 - 参考视频自动生成 (优先级: P1)

Skill 执行成功后，系统自动按照知识库中的技术维度，从原始视频中截取最高置信度的示范片段，叠加技术标注文字（如"肘部角度: 90°~120°"），按维度顺序拼接为一段完整的标准参考视频，上传至 COS 供管理员预览。

**优先级原因**: 参考视频是管理员审批 KB 质量的核心工具——不看视频无法直观判断知识库是否准确，这也是 Feature-003 区别于直接审批文字 KB 的核心差异。

**独立测试**: 执行 Skill 后，访问参考视频 URL，验证视频包含正确的技术维度标注文字，且视频时长与所包含的维度数量正相关。

**验收场景**:

1. **给定** KB 包含 5 个不同技术维度的要点，**当** 参考视频生成完成，**那么** 视频中每个片段都叠加了对应维度的标注文字（白色字幕，黑色背景框）。
2. **给定** 某技术维度的要点有 `conflict_flag=True`，**当** 对应片段出现在参考视频中，**那么** 标注文字包含"⚠ 冲突待审核"黄色警告文字。
3. **给定** 参考视频生成过程中 FFmpeg 命令失败，**当** 查询执行状态，**那么** `reference_video.generation_status` 为 `generation_failed`，但 `execution.status` 仍为 `success`（KB 草稿不受影响）。

---

### 用户故事 3 - 审批与入库 (优先级: P1)

管理员预览参考视频后，可对执行结果进行审批（通过或驳回）。审批通过后，KB 草稿自动变为 active 版本（同时归档旧版本）；驳回后 KB 保持 draft 状态，执行记录标记为 rejected，管理员可修改 Skill 配置后重新执行。

**优先级原因**: 审批机制是 KB 质量门控的关键环节，没有审批机制，任何自动生成的 KB 都无法安全地投入使用。

**独立测试**: 执行 Skill 后，分别测试"审批通过"和"驳回"两条路径，验证 KB 状态变更正确，且驳回路径不影响已有 active KB。

**验收场景**:

1. **给定** execution 状态为 `success`，**当** 管理员调用 approve 接口，**那么** execution 状态变为 `approved`，对应 KB 版本状态变为 `active`。
2. **给定** KB 存在 `conflict_flag=True` 的技术要点，**当** 管理员调用 approve 接口，**那么** 返回 422 错误，error.code 为 `CONFLICT_UNRESOLVED`，提示解决冲突后再审批。
3. **给定** execution 状态为 `success`，**当** 管理员调用 reject 接口并提供驳回原因，**那么** execution 状态变为 `rejected`，rejection_reason 保存到记录中，KB 保持 draft 状态。

---

### 用户故事 4 - Skill 生命周期管理 (优先级: P2)

管理员可对 Skill 进行增删改查管理：创建新 Skill、更新视频来源或配置、软删除不再使用的 Skill。软删除后的 Skill 不可再触发执行，但历史执行记录和 KB 保留。

**优先级原因**: 基础 CRUD 是任何可配置实体的必要能力，P2 是因为它不阻塞核心执行流程。

**独立测试**: 创建 Skill → 更新配置 → 执行 → 软删除 → 尝试再次执行（应返回 404 或 422），验证完整生命周期。

**验收场景**:

1. **给定** 管理员发送 POST /api/v1/skills，**那么** 返回 201 和包含 skill_id 的响应体。
2. **给定** 管理员对同一 name 的 Skill 发送第二次 POST，**那么** 返回 409，error.code 为 `SKILL_NAME_DUPLICATE`。
3. **给定** 管理员对已软删除的 Skill 触发执行，**那么** 返回 404，error.code 为 `SKILL_NOT_FOUND`。

## 功能需求

| ID | 描述 | 来源用户故事 |
|----|------|--------------|
| FR-001 | 创建、读取、更新、软删除 Skill 实体（CRUD） | US4 |
| FR-002 | Skill 支持两种视频来源：COS 前缀 或 已分析任务 ID 列表 | US1 |
| FR-003 | 触发 Skill 执行，创建 SkillExecution 记录，异步处理 | US1 |
| FR-004 | 执行过程中按视频来源批量获取视频并运行 KB 提炼流程 | US1 |
| FR-005 | 提炼完成后自动创建 KB 草稿版本 | US1 |
| FR-006 | KB 草稿创建后，自动触发参考视频生成任务 | US2 |
| FR-007 | 参考视频按维度从原始视频中截取最高置信度片段 | US2 |
| FR-008 | 参考视频片段叠加技术标注文字（含冲突警告） | US2 |
| FR-009 | 参考视频上传至 COS，路径可配置 | US2 |
| FR-010 | 参考视频生成失败不影响 SkillExecution 的 success 状态 | US2 |
| FR-011 | 审批 execution 触发 KB 版本 active（单活版本约束） | US3 |
| FR-012 | 审批前检查 conflict_flag，有冲突则阻断并返回 CONFLICT_UNRESOLVED | US3 |
| FR-013 | 驳回 execution 保存 rejection_reason，KB 保持 draft | US3 |
| FR-014 | Skill 名称全局唯一，重复创建返回 409 | US4 |
| FR-015 | 同一 Skill 可多次执行，每次产出独立 SkillExecution 记录 | US1 |

## 非功能需求

| 类别 | 指标 |
|------|------|
| 性能 | Skill 执行（不含视频分析）触发延迟 < 2s |
| 性能 | 参考视频生成（5 个维度，每段 15s）≤ 5 分钟 |
| 可靠性 | 参考视频生成失败后执行记录状态正确，不丢失 KB 草稿 |
| 可维护性 | 所有 FFmpeg 临时文件在 finally 块中清理 |
| 兼容性 | 不修改任何现有 /tasks/* 和 /knowledge-base/* 端点 |
